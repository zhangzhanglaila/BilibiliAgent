# 运行模式切换说明

这份文档只回答一件事：

- 当前项目进入页面时，为什么会默认是某种模式
- 用户在页面里切换开关后，系统到底会怎么处理

---

## 1. 进入系统时，默认模式怎么决定

当前项目的首屏默认状态，先看 `.env`。

判断规则是：

- 如果 `.env` 里已经有 `LLM_API_KEY`
  - 页面第一次打开时，默认就是 `LLM Agent 模式`
  - 运行模式区域的开关默认是开启状态
- 如果 `.env` 里没有 `LLM_API_KEY`
  - 页面第一次打开时，默认就是 `无 Key 逻辑模式`
  - 运行模式区域的开关默认是关闭状态

对应实现位置：

- [web/app.py](D:/agent/web/app.py)
  - `RUNTIME_LLM_ENABLED = CONFIG.llm_enabled()`
  - `RUNTIME_LLM_CONFIG = ... if CONFIG.llm_enabled() else None`
  - `build_runtime_payload()`
  - `index()`
- [web/templates/index.html](D:/agent/web/templates/index.html)
  - 页面首屏直接使用 `initial_runtime`
- [web/static/app.js](D:/agent/web/static/app.js)
  - `INITIAL_RUNTIME`
  - `state.runtime`

这意味着：

- 页面不是先写死成“关闭”，再异步改成“开启”
- 而是服务端首屏渲染时，就已经按当前后端模式把开关状态、标题、提示文案一起渲染好了

---

## 2. 页面里的开关怎么工作

运行模式区域右侧的开关，对应的是“当前这次运行时要不要启用 LLM Agent 模式”。

### 2.1 从开启切到关闭

效果：

- 当前模式立刻从 `LLM Agent 模式` 变成 `无 Key 逻辑模式`
- 聊天助手立刻禁用
- 内容创作和视频分析后续请求改走规则链路

对应实现：

- 前端：`web/static/app.js` 的 `toggleRuntimeMode()`
- 后端接口：`POST /api/runtime-mode`
- 后端方法：`web/app.py` 的 `api_runtime_mode()`、`set_runtime_llm_enabled(False)`

注意：

- 关闭开关不会清空已保存的 LLM 配置
- 所以如果本来就有 `.env` 配置，或者用户之前已经在页面里提交过一组配置，后面再重新打开开关时可以直接恢复

### 2.2 从关闭切到开启

系统会先判断：当前有没有可复用的 LLM 配置。

如果有配置：

- 直接开启
- 进入 `LLM Agent 模式`
- 不显示输入框

如果没有配置：

- 不会直接开启
- 会在运行模式区域下方显示输入框
- 需要用户填写：
  - `URL`
  - `Key`
  - `模型供应商`
  - `模型名称（选填）`
- 提交成功后，立即开启 `LLM Agent 模式`

对应实现：

- 前端：
  - `web/static/app.js` 的 `toggleRuntimeMode()`
  - `submitRuntimeConfig()`
  - `setRuntimeConfigFormVisible()`
- 后端：
  - `web/app.py` 的 `api_runtime_mode()`
  - `api_runtime_llm_config()`
  - `save_runtime_llm_config()`
  - `sanitize_runtime_llm_config_payload()`

## 3. “有配置”到底指什么

这里的“有配置”有两种来源：

### 3.1 `.env` 配置

如果启动服务时 `.env` 里已经有：

```env
LLM_PROVIDER=openai
LLM_API_KEY=你的真实 Key
LLM_BASE_URL=https://你的网关/v1
LLM_MODEL=你的模型名
```

那么：

- 首屏默认就是开启状态
- 关闭后再次打开，可以直接恢复开启

### 3.2 页面运行时填写的配置

如果启动时 `.env` 没有 Key，但用户后来在页面里填写并提交了：

- URL
- Key
- 模型供应商
- 模型名称

那么：

- 当前服务进程里会把这组配置保存到运行时内存
- 当前服务进程后续再次关闭再开启开关，可以直接复用

注意：

- 这组页面里提交的配置是“运行时配置”
- 它不是自动回写 `.env`
- 如果你重启 Python 服务，默认模式仍然重新按 `.env` 判断

## 4. 智能会话助手在关闭状态下为什么点不了

当当前模式是 `无 Key 逻辑模式` 时：

- 智能会话助手模块会进入禁用态
- 鼠标移上去是禁用光标
- 点击时不会发送请求
- 会触发运行模式开关震动
- 同时弹出提示：
  - `请开启LLM Agent模式才能使用智能会话助手。`

对应实现：

- [web/templates/index.html](D:/agent/web/templates/index.html)
  - `assistantLockOverlay`
- [web/static/style.css](D:/agent/web/static/style.css)
  - `.assistant-panel.is-disabled`
  - `.assistant-lock-overlay`
  - `.mode-switch.is-shaking`
- [web/static/app.js](D:/agent/web/static/app.js)
  - `handleAssistantLockedClick()`
  - `shakeRuntimeModeToggle()`

---

## 5. 当前这版逻辑的最终规则

可以直接按下面这 4 句理解：

1. 启动服务时，如果 `.env` 有 Key，页面首屏默认开关就是开。
2. 启动服务时，如果 `.env` 没有 Key，页面首屏默认开关就是关。
3. 开关从关到开时，先看当前有没有现成配置；有就直接开，没有才显示输入框。
4. 开关从开到关时，立即切回无 Key 逻辑模式，智能助手同时禁用。

## 6. 最短验证方法

### 6.1 验证“有 `.env` 时默认开启”

1. 在 [`.env`](D:/agent/.env) 里填入 `LLM_API_KEY`
2. 启动服务
3. 打开页面
4. 看“运行模式”右侧开关是否默认开启

### 6.2 验证“无 `.env` 时默认关闭”

1. 清空 [`.env`](D:/agent/.env) 里的 `LLM_API_KEY`
2. 启动服务
3. 打开页面
4. 看“运行模式”右侧开关是否默认关闭

### 6.3 验证“关闭后再开启是否复用已有配置”

1. 先让系统处于开启状态
2. 手动关闭开关
3. 再次打开开关
4. 如果当前已有 `.env` 配置或本进程内已保存配置，应直接开启，不应再次弹出输入框

### 6.4 验证“无配置时助手点击是否只提示不放行”

1. 让系统处于关闭状态，且当前没有可复用 LLM 配置
2. 点击右侧智能会话助手区域
3. 观察是否出现：
   - 禁用光标
   - 开关震动
   - 提示“请开启LLM Agent模式才能使用智能会话助手。”
