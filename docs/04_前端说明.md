# 前端说明

## 1. 当前页面结构

当前页面由 3 个区域组成：

- 左侧主工作区
- 右侧智能对话助手
- 顶部运行模式与全局状态区

对应文件：

- `web/templates/index.html`
- `web/static/app.js`
- `web/static/style.css`

页面不是 Tab 切换，而是同屏工作台：

- 模块一：选题与文案
- 模块二：视频解析与优化
- 右侧：聊天式智能助手

## 2. 布局行为

### 桌面端

桌面端左侧两个业务模块使用悬停伸缩布局：

- 默认模块一更宽，模块二更窄
- 鼠标移到模块一，模块一继续放大
- 鼠标移到模块二，模块二放大，模块一缩窄

这样做的目的是让用户在当前操作的模块上获得更大可视区域，但不需要切页面。

### 移动端

移动端自动退化为单列布局：

- 模块一
- 模块二
- 聊天助手

不启用悬停伸缩。

## 3. 模块一：选题与文案

### 输入项

- `creatorField`
- `creatorDirection`
- `creatorIdea`
- `creatorPartition`
- `creatorStyle`

### 前端行为

点击“一键生成选题与文案”后，前端会：

1. 读取输入内容
2. 调用 `/api/module-create`
3. 显示 B 站风格进度条
4. 渲染选题结果与文案结果

### 结果区

模块一结果区会展示：

- 整理后的创作方向
- 推荐主选题
- 热门选题建议
- 标题
- 脚本
- 简介
- 标签
- 置顶评论

## 4. 模块二：视频解析与优化

### 输入项

- `videoLink`

### 自动解析机制

模块二不是点分析后才开始解析。

当前实现是：

1. 用户在 `videoLink` 输入或粘贴链接
2. 前端自动请求 `/api/resolve-bili-link`
3. 预览区立即展示当前视频的结构化空框或真实解析结果
4. 用户再点击“一键解析并分析视频”

### 预览区展示

在真正开始分析前，页面就会显示这些字段的占位卡片：

- 视频标题
- 视频类型 / 分区
- UP 主
- BV 号
- 播放量
- 点赞量
- 投币 / 收藏
- 评论 / 分享

鼠标悬停会显示提示，说明这些字段来自“根据视频链接自动解析”。

### 分析按钮行为

点击“一键解析并分析视频”后：

1. 前端优先使用已自动解析成功的 `resolved`
2. 后端 `/api/module-analyze` 优先复用这个 `resolved`
3. 再做热门 / 低表现判断
4. 返回分析结论、优化建议、参考视频

### 结果区

模块二结果区会展示：

- 视频标题
- 视频分区
- UP 主
- BV 号
- 播放 / 点赞 / 投币 / 收藏 / 评论 / 分享
- 热门 / 待优化判断
- 原因分析
- 优化建议
- 后续选题
- 参考视频跳转链接

## 5. 智能对话助手

### 布局

智能助手固定在页面右侧，采用聊天式布局：

- 对话记录在中间滚动区域
- 输入框固定在底部
- 用户消息在右侧
- 助手回复在左侧

### 对话能力

只有在 LLM 模式下启用：

- `/api/runtime-info` 返回 `chat_available=true`
- 前端解锁输入框和发送按钮
- 点击发送后调用 `/api/chat`

### 聊天结果

每条助手回复支持附加信息：

- `agent_trace`
- `suggested_next_actions`
- `reference_links`

其中 `reference_links` 会渲染成可直接点击跳转的 B 站链接。

## 6. 进度反馈

模块一、模块二、聊天助手都使用统一的进度展示：

- 顶部状态切换为处理中
- 结果区显示 B 站风格进度条
- 同时展示阶段性步骤文本

这样可以避免用户误以为页面卡死。

## 7. 模式识别

前端启动时先调用：

- `/api/runtime-info`

再根据返回结果更新页面：

- 顶部模式 Banner
- Hero 区模式徽标
- 右侧助手可用状态
- Token 提示

### 逻辑链路

- `mode=rules`
- 模块一和模块二都走纯代码逻辑
- 聊天助手禁用

### LLM 链路

- `mode=llm_agent`
- 模块一和模块二走 `LLMWorkspaceAgent`
- 聊天助手启用

## 8. 兼容接口

虽然页面现在是“两个业务模块 + 一个聊天助手”，但这些兼容接口仍保留：

- `/api/topic`
- `/api/copy`
- `/api/operate`
- `/api/optimize`
- `/api/pipeline`

这样可以继续兼容原有项目结构，不需要破坏已有代码组织。
