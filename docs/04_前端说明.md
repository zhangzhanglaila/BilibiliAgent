# 前端说明

## 1. 当前页面结构

当前前端由 3 个大区域组成：

- 顶部状态区
- 左侧业务工作区
- 右侧智能对话助手

对应文件：

- `web/templates/index.html`
- `web/static/app.js`
- `web/static/style.css`

## 2. 当前页面不是旧版双面板悬停布局

这里必须明确说明，避免继续沿用旧文档描述。

当前真实交互是：

- 左侧顶部有两个标签按钮
  - `视频分析`
  - `内容创作`
- 点击标签切换左侧主模块
- 右侧智能助手始终存在

也就是说：

- 不是旧文档中的“同屏双模块悬停伸缩主布局”
- 当前代码里 `setActiveModule()` 控制的是模块切换
- `#moduleGrid` 通过 `data-active="analyze/create"` 控制显示状态

## 3. 顶部状态区

顶部状态区负责显示全局工作状态和运行模式。

当前包含：

- 全局状态胶囊 `#globalStatusPill`
- 状态文字 `#statusText`
- 清空结果按钮 `#clearResultsBtn`
- Hero 区模式徽标 `#runtimeModeBadge`
- 运行模式说明面板 `#runtimeModePanel`

运行模式信息不是前端硬编码写死的，而是前端初始化时请求：

- `GET /api/runtime-info`

然后用返回内容更新：

- `mode`
- `mode_label`
- `mode_title`
- `mode_description`
- `token_policy`
- `switch_hint`
- `chat_available`

## 4. 左侧业务工作区

左侧业务工作区有两个模块：

- 视频分析
- 内容创作

通过顶部标签切换。

### 4.1 视频分析模块

对应 DOM：

- `#videoLink`
- `#videoPreview`
- `#videoAnalyzeBtn`
- `#videoResult`

核心前端逻辑：

1. 用户在 `#videoLink` 输入或粘贴链接
2. `input` 事件触发 `scheduleVideoResolve()`
3. 前端延迟约 550ms 后调用 `resolveVideoLink()`
4. 请求 `/api/resolve-bili-link`
5. 先在 `#videoPreview` 里显示“正在自动解析”
6. 成功后缓存到：
   - `state.videoResolved`
   - `state.videoResolvedUrl`
7. 点击“分析”按钮时再调用 `/api/module-analyze`

### 4.2 内容创作模块

对应 DOM：

- `#creatorField`
- `#creatorDirection`
- `#creatorIdea`
- `#creatorPartition`
- `#creatorStyle`
- `#creatorRunBtn`
- `#creatorResult`

核心前端逻辑：

1. 读取领域、方向、想法、分区、风格
2. 调用 `/api/module-create`
3. 渲染整理后的创作方向、核心问题、主选题、选题卡片和文案

## 5. 视频分析模块的真实行为

### 5.1 先自动解析，再分析

当前实现不是“点击分析按钮后才开始解析”。

而是：

- 链接一输入就先自动解析
- 预览区先展示真实视频信息或错误提示
- 分析按钮只负责进一步调用 `/api/module-analyze`

### 5.2 视频预览区展示什么

预览区主要展示：

- 视频标题
- 视频类型 / 分区
- UP 主
- BV 号
- 播放量
- 点赞量
- 投币量
- 收藏量
- 评论量
- 分享量

如果有封面，还会显示封面条带区域。

### 5.3 分析结果区展示什么

分析结果区会根据返回内容渲染：

- 表现判断卡片
- 核心指标
- 原因分析
- 后续选题
- 优化建议
- 低表现时的新文案
- 参考视频卡片

其中参考视频来自：

- 当前市场样本
- 当前视频相关推荐
- B 站搜索结果

## 6. 内容创作模块的真实行为

### 6.1 输入校验

内容创作模块允许 3 个输入字段至少填 1 个：

- 领域
- 方向
- 想法

如果三者都为空，前端会直接提示，不发请求。

### 6.2 结果区展示什么

返回结果会渲染：

- 整理后的方向 `normalized_profile`
- 当前核心问题 `seed_topic`
- 推荐主选题 `chosen_topic`
- 文案风格
- 3 个选题建议卡片
- 标题区
- 脚本区
- 简介区
- 标签区
- 置顶评论区

## 7. 智能对话助手

### 7.1 当前布局

智能助手固定在页面右侧，包含：

- 对话线程区 `#assistantResult`
- 快捷提问按钮
- 多行输入框 `#assistantMessage`
- 发送/语音按钮 `#assistantSendBtn`

### 7.2 当前能力

只有在 `chat_available=true` 时启用。

启用后支持：

- 连续对话
- 快捷提问
- 上下文感知
- 可点击参考视频链接
- 打字机输出效果
- 浏览器语音输入

### 7.3 输入上下文

助手并不是只拿用户一句话。

前端每次会一起提交页面上下文：

```json
{
  "field": "",
  "direction": "",
  "idea": "",
  "partition": "",
  "style": "",
  "videoLink": ""
}
```

所以它会结合：

- 当前创作输入
- 当前视频链接
- 历史消息

一起理解问题。

### 7.4 快捷提问按钮

当前固定有 4 个快捷按钮：

- 帮我选题
- 帮我分析视频
- 帮我写文案
- 看最近热点

如果当前是规则模式：

- 点击快捷按钮只会把文字填进输入框

如果当前是 LLM 模式：

- 点击快捷按钮会直接发送

### 7.5 打字机回复

当前助手回复不是一次性整块渲染，而是：

1. 先显示 pending 气泡
2. 请求 `/api/chat`
3. 收到结果后，把 `reply` 放进 `fullContent`
4. 通过 `typeAssistantReply()` 逐字渲染

### 7.6 语音输入

当前前端支持浏览器语音输入：

- `SpeechRecognition`
- `webkitSpeechRecognition`

如果浏览器不支持，会提示用户直接输入文字。

## 8. 复制能力

当前前端大量使用 `data-copy` 机制统一处理复制按钮。

支持复制的内容包括：

- 标题集合
- 单个标题
- 完整脚本
- 单个脚本片段
- 视频简介
- 置顶评论
- 助手回复
- 优化标题

复制逻辑统一由：

- `copyText()`
- `bindCopyButtons()`

负责。

## 9. 封面加载和 fallback

当前前端不是简单直接放图片。

封面组件支持：

- 统一封面框架 `cover-frame`
- 懒加载
- 加载中状态
- 自动重试
- fallback 占位

当前逻辑包含：

- 原始封面地址标准化
- 最多 2 次重试
- retry query 参数防缓存
- 最终失败后回退为占位块

## 10. 进度反馈

当前 3 条主流程都用了统一的进度反馈机制：

- 内容创作
- 视频分析
- 聊天助手

具体表现为：

- 顶部状态切换
- 区块内百分比进度条
- 阶段步骤文案

这不是后端真实进度，而是前端的平滑进度体验。

## 11. 清空结果按钮

当前顶部有全局清空按钮：

- `#clearResultsBtn`

点击后会：

- 清空内容创作结果
- 清空视频分析结果
- 清空视频预览缓存
- 清空聊天记录
- 停止所有进度 job

## 12. 响应式布局

当前样式表已经包含多段 `@media` 规则。

也就是说：

- 桌面端和移动端都考虑了布局适配
- 不是只做了桌面版

当前可从 `style.css` 中看到的断点包括：

- `max-width: 1240px`
- `max-width: 1180px`
- `max-width: 980px`
- `max-width: 640px`

## 13. 前后端接口映射

前端当前真实使用的主接口如下：

- `GET /api/runtime-info`
- `POST /api/resolve-bili-link`
- `POST /api/module-create`
- `POST /api/module-analyze`
- `POST /api/chat`

前端没有直接调用兼容接口：

- `/api/topic`
- `/api/copy`
- `/api/operate`
- `/api/optimize`
- `/api/pipeline`

这些是为兼容旧结构和 CLI/接口复用保留的。
