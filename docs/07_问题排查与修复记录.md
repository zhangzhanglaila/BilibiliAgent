# 问题排查与修复记录

本文记录当前项目里已经确认过的一批真实问题和修复方式，方便后续排查时直接回看。

## 1. 运行时 LLM 配置缺失导致页面反复报错

### 现象

- 页面在 LLM 模式下分析失败
- 用户只看到连接错误，不知道去哪里补 URL / Key / provider / model

### 修复

- 增加运行时配置保存接口
- 页面内直接展示配置表单
- 失败后可自动拉起重配入口

## 2. B 站短链 / 跟踪链接 / av 链接解析不稳定

### 现象

- `b23.tv`
- `cm.bilibili.com`
- 旧版 `av` 链接

不能稳定进入统一解析链路。

### 修复

- 在解析入口统一展开和归一
- 提取 BV / av
- 构建统一 `resolved` 结果

## 3. 文案标题长期掉回提问句模板

### 现象

- 标题反复出现提问句
- 容易出现“为什么 / 怎么 / 哪种更容易起量”这类固定模板

### 修复

- 在 `CopywritingAgent` 和 `OptimizationAgent` 里增加标题清洗规则
- 对生活区 / 恋爱日常类脚本做更强约束

## 4. 前端文档和页面真实结构不一致

### 现象

- 文档长期写“双模块 + 聊天”
- 但当前页面实际已经是三模块

### 修复

- 统一改为：
  - 视频分析
  - 内容创作
  - 知识库管理
  - 右侧智能助手

## 5. 知识库检索把同一篇文档多个分片当多条结果显示

### 现象

- 搜索结果里会看到同一篇文档重复出现
- 有时甚至直接显示出后半段字段，例如 `"链接": "https://..."` 这种片段

### 根因

- Chroma 返回的是分片命中
- 展示层直接按分片逐条渲染，没有按 `document_id` 合并

### 修复

- 搜索结果渲染前先按文档去重
- 优先保留主片段内容

## 6. 知识库结果标题直接显示内部文档 ID

### 现象

- 标题会显示：
  - `分区热门榜:tech:BV...`
  - `全站热门榜:BV...`

### 根因

- 展示层回退逻辑直接把 `document_id` 当标题

### 修复

- 标题优先提取 `视频标题`
- 热门样本显示为“榜单名 + 分区 + 视频标题”
- 不再把原始 `BV...` 文档 ID 当主标题

## 7. 一级分类检索误召回

### 现象

- 点击 `舞蹈` 时，搜出“翻唱”“不知火舞”等不属于舞蹈分类的内容
- 甚至搜出 `分区热门榜:knowledge` 这类不相关样本

### 根因

- 旧逻辑只是拿分类词做纯向量检索
- 没有在召回后做分类过滤

### 修复

- 搜索先扩召回
- 再按一级分类规则做过滤
- 最后按文档去重

## 8. 热门样本同步只抓 5 个内部粗分区

### 现象

- 知识库里有 `分区热门榜:knowledge / tech / life / game / ent`
- 但用户在 `popular/rank` 页面看到的 `国创 / 舞蹈 / 音乐` 等真实分区没有单独同步

## 9. AI Key 过期后项目调不到模型

### 现象

- 页面切到 `LLM Agent 模式` 后直接报鉴权失败
- 聊天、视频分析、内容创作都退不出来结果
- 原因通常是旧 `LLM_API_KEY` 失效，或者 provider / base_url / model 已经换了

### 修复

- 项目实际读取的是根目录 `.env`
- 关键字段是：
  - `LLM_PROVIDER`
  - `LLM_API_KEY`
  - `OPENAI_API_KEY`
  - `LLM_BASE_URL`
  - `LLM_MODEL`
  - `LLM_REASONING_EFFORT`
  - `LLM_DISABLE_RESPONSE_STORAGE`
- 改完后重启 `python web/app.py`

### 备注

- 页面内运行时配置只适合临时改 `provider / url / key / model`
- 如果想把策略长期固定下来，还是改 `.env`
- 当前项目已支持把 `LLM_REASONING_EFFORT` 透传给模型调用

### 根因

- 旧实现只抓项目内部主分区

### 修复

- 改成抓真实一级分区榜
- 同步前清理旧的粗分区榜样本
- 给热门样本追加 `榜单链接`

## 9. 热门知识库同步全部失败：`invalid literal for int() with base 10: '0:47'`

### 现象

- 点击“自动更新热门知识库”后全部失败
- 报错里出现：
  - `invalid literal for int() with base 10: '0:47'`

### 根因

- 热门样本里部分 `duration` 不是纯数字，而是 `0:47` 这类字符串
- 旧逻辑直接 `int(duration)`，导致整条样本失败

### 修复

- 增加时长解析函数
- 兼容：
  - `0:47`
  - `01:23`
  - `1:02:03`
  - 数字秒数

## 10. 热门知识库同步全部失败：`title` 未定义

### 现象

- 同步更新后所有热门样本都失败

### 根因

- 给热门样本 metadata 增加 `title` 字段时，直接引用了未定义变量
- 这个错误属于运行时错误，不会在简单编译检查时暴露

### 修复

- 在 `_ingest_hot_items()` 内先显式提取标题
- 再写入 metadata

## 11. 当前环境缺少 Chroma 依赖导致知识库不可写

### 现象

- 点击知识库上传、查看、检索、同步时直接失败
- 或状态里显示 `Chroma backend not initialized`

### 根因

- 当前运行环境没装：
  - `chromadb`
  - `langchain-chroma`
- 旧实现遇到这种情况时，知识库会整体不可用

### 修复

- 知识库后端改成：
  - 优先使用 Chroma
  - 缺依赖时自动回退到本地 JSON 存储
- 状态区额外显示当前后端类型和回退说明

## 12. JSON fallback 文件里为什么会看到向量数据

### 现象

- 打开 `vector_db/bilibili_knowledge__fallback_store.json`
- 发现每条记录里除了正文，还有一大段 `embedding`
- 容易误以为“系统把整份向量 JSON 直接发给了大模型”

### 根因

- 这份文件是知识库 fallback 存储，不是单独导出的“原始热门样本 JSON”
- 每条记录当前同时保存：
  - `text`
  - `metadata`
  - `embedding`
- 其中：
  - `text` 才是实际可读的结构化业务内容
  - `metadata` 是过滤和展示辅助字段
  - `embedding` 只用于检索排序

### 修复 / 当前真实行为

- 当前 Agent 或知识库接口不会把整份 fallback 文件直接发给大模型
- 当前检索链路只会把命中的：
  - `id`
  - `text`
  - `metadata`
  - `score`
  返回给上层
- 大模型真正看到的是命中结果，不是整库，也不是整段向量数组
- 页面里的“关键词检索”接口本身也不会调用大模型，只是后端检索后返回前端展示

## 15. LangSmith Trace 反馈问题：`module-analyze` 上下文污染与 ReAct 工具调用失效

### 现象

- 调用 `/api/module-analyze` 分析赶海视频时，输出混入了和当前视频无关的全站热榜内容。
- LangSmith Trace 中出现 `agent_trace: []`、`tool_observations: []`，说明模型直接输出了 final，没有真实调用工具。

### 先说结论

这个问题和最初的判断只部分一致。

当前项目里，`creator_briefing` 预加载实际上已经只在 `module-create` 中按关键词触发，并不会直接在 `module-analyze` 中执行。分析模块被污染的真实来源是：

1. `/api/module-analyze` 在进入 Agent 前预先构造了完整 `market_snapshot`。
2. `video_briefing` 工具内部过去也会携带全站热榜和分区样本。

也就是说，问题核心不是“创作预处理串到了分析模块”，而是“分析模块默认上下文过满，导致 ReAct 没有真正开始工作”。

### 根因

#### 根因 1：预加载上下文和 ReAct 工具边界混乱

旧实现中，分析模块在进入 `LLMWorkspaceAgent.run_structured()` 之前，就先把较完整的市场样本塞进了 `user_payload`。

这会带来两个后果：

1. 模型误判自己已经拿到了足够多的信息。
2. 这些信息没有经过工具调用，因此不会出现在 LangSmith Trace 里。

#### 根因 2：分析链路缺少最低工具调用约束

如果预加载 payload 过满，同时 `required_tools` 没有明确约束，模型就很容易直接输出 final，导致：

- `retrieval` 没有调用
- `video_briefing` 没有调用
- `agent_trace` 为空
- `tool_observations` 为空

### 修复方案

#### 1. 创作模块预加载保持收敛

`creator_briefing` 仍然只在 `module-create` 中按关键词触发，关键词包括：

- `选题`
- `赛道`
- `分区`
- `热点`
- `竞品`
- `爆款`

#### 2. 分析模块默认不再预加载全站热榜

新增 `build_empty_market_snapshot()`，让 `module-analyze` 进入 Agent 时只携带最小化上下文：

- `hot_board = []`
- `partition_samples = []`
- `peer_samples = []`
- `source_count = 0`

这样分析模块默认只带：

- 当前视频的 `resolved`
- 空的 `market_snapshot`

不再在主循环前塞入全站热榜样本。

#### 3. `video_briefing` 只返回轻量样本

新增 `build_peer_only_market_snapshot()`，将 `video_briefing` 的返回范围收敛为：

- 当前视频元数据
- 可选的同 UP 主 `peer_samples`

明确不再返回：

- `hot_board`
- `partition_samples`

这样即使模型调用 `video_briefing`，也不会再次把全站热榜内容混进上下文。

#### 4. 分析模块强制先调工具再 final

在 `run_llm_module_analyze()` 中显式传入：

- `required_tools=("video_briefing", "retrieval")`
- `strict_required_tool_order=True`

这意味着分析模块在输出 final 之前，必须真实调用过这两个工具。

如果模型试图直接输出 final，Agent 会记录校验错误 observation，并继续循环，而不是直接返回。

#### 5. 强化 ReAct Prompt

`LLMWorkspaceAgent` 的系统提示词增加硬约束：

- 必须首先调用工具获取信息
- 严禁直接输出 final
- 如果当前任务要求 `video_briefing + retrieval`，必须先调用 `video_briefing`，再调用 `retrieval`

同时在 user prompt 中补充规则：

- 当 `required_tools` 非空时，在这些工具真正调用完成之前，禁止直接输出 final

#### 6. fallback 提示词同步纠偏

分析链路的 fallback prompt 过去还写着“已经拿到同类样本”，这和新的最小化预载策略不一致。

本次已改成：

- 当前只拿到真实视频信息和最小化市场上下文
- 市场样本可能为空
- 不包含全站热榜预加载数据

这样即使 Agent 回退到 direct LLM fallback，也不会被错误提示误导。

### 修复后的预期行为

修复后的 `module-analyze` 链路应为：

1. 后端先解析用户输入视频，得到 `resolved`
2. 后端只注入空的 `market_snapshot`
3. 进入 `LLMWorkspaceAgent.run_structured()`
4. Agent 约束模型必须先调用 `video_briefing`
5. Agent 再约束模型必须调用 `retrieval`
6. 模型在拿到 observation 后才能输出 final
7. 只有当 `retrieval` 返回样本不足时，模型才允许继续调用 `web_search`

### 验证标准

重新运行 `module-analyze` 后，LangSmith Trace 应满足：

1. `agent_trace` 不为空
2. 至少包含 `retrieval` 和 `video_briefing`
3. `tool_observations` 不为空
4. `tool_observations` 中包含工具真实返回的数据
5. 默认输出只围绕用户输入视频，不再混入未调用得到的全站热榜内容

### 基于 2026-04-03 21:35 LangSmith 的实测链路补充

本次对 `docs/test/langsmith记录202604032135.md` 的实测确认了两件事：

1. 当前代码里的强制工具顺序，已经是 `video_briefing -> retrieval`，不是旧文档里写过的 `retrieval -> video_briefing`。
2. 这次 trace 的主链路已经真实跑通工具，但中间仍出现了一次“模型想重复调 `video_briefing`，被预算护栏拦下”的额外回合。

#### 实际调用链路

按这份 LangSmith 导出，实际发生的链路是：

```text
web.run_llm_module_analyze
-> llm_workspace_agent.run_structured
   -> llm_client.invoke_messages #1
      -> ChatOpenAI
      -> 返回 action=video_briefing
   -> agent_tool.video_briefing
      -> build_video_briefing()
      -> build_peer_only_market_snapshot()
   -> llm_client.invoke_messages #2
      -> ChatOpenAI
      -> 返回 action=retrieval
   -> agent_tool.retrieval
      -> knowledge_base.retrieve
   -> llm_client.invoke_messages #3
      -> ChatOpenAI
      -> 返回 action=video_briefing
   -> validation_error
      -> 原因：video_briefing 已达到 tool limit=1，被 Agent 护栏拦截
   -> llm_client.invoke_messages #4
      -> ChatOpenAI
      -> 返回 action=final
   -> llm_client.invoke_messages #5
      -> ChatOpenAI
      -> _reflect_final 结果质检与必要重写
-> finalize_module_analyze_result
```

可以看成：

- 真实工具调用共 `2` 次：`video_briefing`、`retrieval`
- 被拦截的无效工具提议 `1` 次：重复调用 `video_briefing`
- 最终结果生成 `1` 次
- 反思质检 `1` 次

#### 当前代码预期逻辑

当前代码设计上的预期链路，应当是：

```text
web.run_llm_module_analyze
-> 注入 resolved + 空 market_snapshot
-> llm_workspace_agent.run_structured
   -> 先调 video_briefing
   -> 再调 retrieval
   -> 如果 retrieval 样本不足，才允许 web_search
   -> 直接输出 final
   -> enable_reflection=True 时再做一次 _reflect_final
-> finalize_module_analyze_result
```

和实测相比，差异只有一处：

- 代码预期是 `video_briefing -> retrieval -> final -> reflection`
- 这次实测是 `video_briefing -> retrieval -> 重复 video_briefing 被拦截 -> final -> reflection`

这说明当前问题已经不是“工具没调起来”，而是“模型在拿到 retrieval 结果后，仍偶尔想重复索取已拿过的视频简报”；不过运行时护栏已经生效，所以不会破坏主链路。

#### LangSmith 导出里的层级重复关系

这份导出里，`input / output` 确实有明显重复，但重复程度分层不同：

1. `web.run_llm_module_analyze`
   - 这是最上层业务结果包。
   - `outputs` 里已经包含最终页面要用的结果。
   - 同时也已经带上了 `agent_trace`、`tool_observations`、`reference_videos`。
   - 如果你只想看“最终分析结果是什么、工具有没有真的调用、每个工具回了什么”，这一层通常已经够用。

2. `llm_workspace_agent.run_structured`
   - 这层和顶层有大量重复。
   - 它额外更适合看 Agent 运行时约束，例如：
   - `allowed_tools`
   - `required_tools`
   - `required_final_keys`
   - `system_prompt_override`
   - `user_payload`
   - 它的 `outputs.tool_observations` 里已经把工具真实返回完整保留下来了，信息量通常比单独的 `agent_tool.*` 小节更高。

3. `llm_client.invoke_messages`
   - 这层会重复多轮 `system_prompt` 和 `user_prompt`。
   - 但它是分析“模型为什么做出某次 action 决策”的最关键层。
   - 因为这里能看到：
   - 当前轮完整 prompt
   - 已经使用的工具列表
   - 历史 observation 拼接后的 scratchpad
   - 模型这轮返回的 `content_preview`

4. `ChatOpenAI`
   - 这层和同轮 `llm_client.invoke_messages` 高度重复。
   - `llm_client.invoke_messages.inputs.system_prompt/user_prompt`，本质上就是 `ChatOpenAI.inputs.messages` 的上一层封装。
   - `llm_client.invoke_messages.outputs.content_preview`，本质上也是 `ChatOpenAI.outputs.generations` 里文本内容的摘要。
   - 正常排查业务逻辑时，通常二选一即可，没必要两层都复制。

5. `agent_tool.video_briefing` / `agent_tool.retrieval` / `knowledge_base.retrieve`
   - 这些属于更底层的工具执行节点。
   - 它们和 `tool_observations` 也有重复。
   - 其中：
   - `agent_tool.video_briefing` 只有 `observation_preview`，反而不如 `tool_observations` 完整。
   - `knowledge_base.retrieve` 主要只保留了 `query` 和 `match_count`，远不如 `tool_observations` 里的 `matches` 完整。

#### 以后复制 LangSmith 的建议

以后你不用默认把所有枝条全复制。

推荐按目的选：

- 如果你只是想分析“最终结果对不对、工具有没有调用、工具回了什么”，通常只复制 `web.run_llm_module_analyze` 就够了。
- 如果你要分析“为什么 Agent 走了这个顺序、为什么被拦截、为什么没有按预期出 final”，复制 `web.run_llm_module_analyze + llm_workspace_agent.run_structured + 对应几轮 llm_client.invoke_messages`。
- 如果你怀疑的是 SDK 层、模型供应商层、消息封装层，再额外复制 `ChatOpenAI`。
- 一般不建议把所有 `ChatOpenAI`、`agent_tool.*`、`knowledge_base.retrieve` 全量一起贴，因为大部分内容已经在上层被重复展开过了。

一句话总结：

- 看业务结果：复制最上层 `web.run_llm_module_analyze`
- 看 Agent 决策：再加 `llm_workspace_agent.run_structured`
- 看模型每轮为什么这么选：再加对应的 `llm_client.invoke_messages`

### 根因

#### 根因 1：后处理对 LLM 返回字段形状假设过强

  - 字符串数组
  - 半结构化 JSON 文本

#### 根因 2：fallback 缺少足够强的规则基线护栏

- 主链路崩溃后进入 `run_llm_module_analyze_fallback()`

- 导致高表现视频也可能被模型错误判成低表现

#### 根因 3：trace 命名和函数定义残留脏状态

- `run_llm_module_analyze()` 之前存在错误 trace 装饰痕迹
- `web/app.py` 中还残留了重复的 `get_llm_workspace_agent()` 定义
- 这不会直接制造这次 `ValueError`，但会干扰 LangSmith 归因和后续维护

#### 6. 清理视频分析链路里的 trace 和重复定义问题

### 6. `run_structured()` 里什么时候会调用 LLM

在 `LLM Agent 模式` 下，这一层才是真正的“多轮 LLM 调度器”。

#### 6.1 每一轮决策都会调用一次 LLM

每轮都会把下面这些内容拼成提示词送给 LLM：

- `system_prompt`
- `task_name`
- `task_goal`
- `user_payload`
- 可用工具列表
- 必调工具列表
- 历史 `scratchpad / tool_observations`
- 最终响应契约

LLM 每轮必须只返回固定 JSON 结构：

```json
{
  "action": "工具名 或 final",
  "action_input": {},
  "final": null
}
```

#### 6.2 典型成功链路里的 LLM 调用次数

现在的典型链路一般是：

1. 第 1 轮：决定调用 `retrieval`
2. 第 2 轮：决定直接输出 `final`，或者在样本不足时决定调用 `web_search`
3. 第 3 轮：如果用了 `web_search`，再决定输出 `final`
4. 最后一轮：`_reflect_final()` 做结果质检

也就是说：

- 不走 `web_search` 时，一般是 `2 轮决策 + 1 轮反思`
- 走 `web_search` 时，一般是 `3 轮决策 + 1 轮反思`

### 7. LLM 最终出结果时的输入和输出

当 `retrieval`、`web_search` 等 observation 已经进入 `scratchpad` 后，某一轮 LLM 会开始尝试给出 `final`。

LLM 要输出的 `final` 仍然必须满足：

```json
{
  "resolved": { "...必须有..." },
  "performance": { "...必须有..." },
  "topic_result": { "...必须有..." },
  "optimize_result": { "...必须有..." },
  "copy_result": { "...低表现时必须可直接落地..." },
  "analysis": {
    "analysis_points": [],
    "benchmark_analysis": {},
    "remake_script_structure": {},
    "advanced_title_sets": {},
    "cover_plan": {},
    "tag_strategy": {},
    "publish_strategy": {},
    "reusable_hit_points": []
  }
}
```

注意：

- 虽然 LLM 会返回 `resolved`
- 但真正最终生效的 `resolved` 会在后处理阶段被后端强制覆盖为入口预解析得到的那份

### 8. 反思质检层：`_reflect_final()`

这是主链里第二次明确调用 LLM 的地方。

它会做的事：

1. 检查候选 `final` 是否满足任务要求
2. 如果满足，返回 `pass=true`
3. 如果不满足，返回 `pass=false` 并给出 `rewritten_final`

这一层不是工具调用，而是出站前的质量复核。

### 9. 后处理层：`finalize_module_analyze_result()`

这一层不调用 LLM，全部是代码归一化。

输入：

- `result`：来自 `run_structured()` 或 fallback 的最终结果
- `resolved`：入口预解析结果
- `market_snapshot`：入口预抓的同方向爆款快照

这一层具体做的事：

1. 强制用入口层的 `resolved` 覆盖结果里的 `resolved`
2. 用 `classify_video_performance(resolved)` 做规则基线，归一化 `performance`
3. 归一化 `topic_result`
4. 归一化 `optimize_result`
5. 归一化 `analysis`
6. 如果视频是高表现，强制 `copy_result = null`
7. 如果视频是低表现，归一化 `copy_result`
8. 从 `tool_observations.retrieval.matches` 和 `market_snapshot.peer_samples` 共同提炼 `reference_videos`
9. 补齐 `runtime_mode`

最终输出结构大致是：

```json
{
  "resolved": { "...后端真实视频信息..." },
  "performance": { "...归一化后..." },
  "topic_result": { "...归一化后..." },
  "optimize_result": { "...归一化后..." },
  "copy_result": { "...或 null..." },
  "analysis": { "...归一化后..." },
  "reflection_issues": ["...可选..."],
  "agent_trace": ["retrieval"],
  "tool_observations": [
    {
      "action": "retrieval",
      "action_input": { "query": "...", "limit": 8 },
      "observation": { "...工具结果..." }
    }
  ],
  "runtime_mode": "llm_agent",
  "reference_videos": [
    { "...对标视频..." }
  ]
}
```

如果中间用了 `web_search`，`agent_trace` 里会追加 `web_search`。

### 10. fallback 模式里什么时候调用 LLM

如果 `run_llm_module_analyze()` 内部抛错，入口层会判断是否允许 fallback。

允许 fallback 时，会进入：

- `run_llm_module_analyze_fallback(data, resolved, market_snapshot)`

fallback 只调用一次 LLM，不再走 ReAct 多轮工具链。

输入给 LLM 的内容是：

- `resolved`：当前视频真实信息
- `baseline_performance`：规则基线判断结果
- `market_snapshot`：入口层传入的同方向爆款快照
- 一份要求 LLM 直接输出完整 JSON 的强约束 prompt

然后同样还会进入：

- `finalize_module_analyze_result()`

### 11. 无 Key 逻辑模式下的规则链路

如果没有开启 `LLM Agent 模式`，则整条视频分析链路完全不调用 LLM。

执行顺序是：

1. `resolve_video_payload(url)` 生成 `resolved`
2. `run_topic()` 生成选题方向
3. `classify_video_performance(resolved)` 做表现判断
4. 如果是高表现视频：
   - `build_hot_analysis()`
5. 如果是低表现视频：
   - `build_rule_optimization_agent().run()`
   - `build_rule_copy_agent().run()`
   - `build_low_performance_analysis()`
6. `select_reference_videos()` 生成 `reference_videos`
7. 直接返回结果给前端

### 12. 当前最容易混淆的三个点

#### 12.1 `resolved` 和 `preloaded_context.video` 不是两次不同解析

- `resolved`：入口层已经算好的后端真实视频信息
- `preloaded_context.video`：把同一份 `resolved` 重排成更适合 LLM 读的结构

#### 12.2 `peer_samples` 不是同 UP，也不是 LLM 补出来的

- `peer_samples`：代码抓到的同方向爆款对标视频
- LLM 只是读取这些样本做判断，不负责生成这些样本

#### 12.3 `video_briefing` 仍然存在，但不属于当前分析主链

- 它还可以给其他 Agent 场景复用
- 但 `module_analyze` 现在不会再让 LLM 决定要不要调它

### 13. 当前代码下“什么时候会调用 LLM”总结

#### 13.1 `LLM Agent 模式`

会调用 LLM 的地方有：

1. `run_structured()` 里的每一轮动作决策
2. `run_structured()` 里输出 `final`
3. `_reflect_final()` 结果质检
4. 如果主链失败且允许 fallback，还会有一次 `run_llm_module_analyze_fallback()` 的单次 LLM 调用

#### 13.2 `无 Key 逻辑模式`

- 完全不调用 LLM

### 14. 典型主链路时序图

```text
POST /api/module-analyze
  -> resolve_video_payload(url)                 # 纯代码，得到 resolved
  -> build_hot_peer_market_snapshot(resolved)  # 纯代码，得到同方向爆款 peer_samples
```

### 问题定义

这次要修的不是“工具没调起来”，而是两个架构层面的偏差：

1. 当前视频在 HTTP 入口已经拿到了稳定的 `resolved`，但分析 Agent 里又让模型自己决定是否调用 `video_briefing`，造成重复解析

- 对标样本容易被低表现 UP 自己的历史视频污染
- LangSmith 里会出现“主链已通，但模型还在尝试再拿一次 `video_briefing`”的多余动作

### 为什么不符合当前业务预期

当前业务目标是：

- 当前视频的真实信息应该由后端代码一次性确定
- 对标样本应该是同方向、同主题、同关键词的爆款视频

#### 19.1 `video_briefing` 不应该由分析 Agent 决定是否调用

原因：

- 解析当前视频链接是确定性逻辑
- 入口层已经拿到 `resolved`
- 再让 Agent 决策一次，只会增加链路长度和重复调用概率

#### 19.2 `peer_samples` 不应该默认取同 UP

原因：

- 同 UP 样本只能说明这个账号最近发了什么
- 不能说明同赛道里什么内容在赢
- 如果这个 UP 整体表现低，拿它自己历史视频做对标会把分析方向带偏

本次修法分三部分：

#### 19.3 架构修正：把当前视频上下文前置

改动：

1. 从 `module_analyze` Agent 的 `allowed_tools / required_tools` 中移除 `video_briefing`
2. 在 `run_structured()` 启动前，先由代码完成：
   - `build_hot_peer_market_snapshot(resolved)`
   - `build_video_analyze_preloaded_context(resolved, url, market_snapshot)`
3. Agent 现在只允许决策：
   - `retrieval`
   - `web_search`

效果：

- 当前视频只解析一次
- LLM 不再有机会重复决定“再拿一遍当前视频”

#### 19.4 样本修正：把 `peer_samples` 改成同方向爆款

改动：

1. 新增 `TopicAgent.fetch_hot_peer_videos()`

3. 通过 B 站搜索接口拿候选视频
4. 通过公开视频详情接口补真实指标
5. 过滤规则：

   - 近 `30` 天

   - 点赞 `>= 1000`
   - 最多保留 `6` 条

效果：

- `peer_samples` 现在表示“同方向爆款对标视频”
- 不再默认被同 UP 的低表现视频污染

#### 19.5 工具语义修正：保留 `video_briefing`，但降级为其他场景可复用工具

改动：

- `video_briefing` 没有被删除
- 但它不再属于 `module_analyze` 主链
- 即便在其他场景调用，它返回的 `peer_samples` 也已经改成同方向爆款对标样本

效果：

- 保留工具复用能力
- 又不会再让分析主链受它干扰

### 改动后的真实调用顺序

现在的 `LLM Agent 模式` 主链是：

## 21. `/api/resolve-bili-link` 因压缩页面和补充逻辑耦合导致基础信息解析失败

### 记录时间

- 2026-04-05 10:31:00 +08:00

### 现象

视频分析模块里把 B 站链接复制进搜索框后，前端预览阶段直接报错，典型报错如下：

- `链接解析失败（BV=BV1dcX5BYESE）`
- `public api: <urlopen error _ssl.c:989: The handshake operation timed out>`
- `bilibili_api:`
- `html: 网页源码解析失败: https://www.bilibili.com/video/BV1dcX5BYESE?...: 网页源码中未找到视频标题`

这类报错最容易让人误判成“BV 没提出来”或者“B 站接口整体挂了”，但实际不是。

### 根因

这次问题拆开后有两个独立根因：

#### 21.1 HTML 兜底解析没有正确处理 B 站压缩响应

- B 站视频页当前会返回 `Content-Encoding: gzip`
- 旧版 `fetch_text()` 直接把响应体按 UTF-8 解码，没有先解压
- 结果就是 HTML 兜底层拿到的是乱码字节流
- 后面的 `extract_initial_state()`、`og:title`、`<title>`、`"title"` 正则都会失效
- 最终表现成“网页源码中未找到视频标题”

#### 21.2 主信息获取和附加补充逻辑绑在同一个失败域里

`fetch_video_info()` 原来虽然先调了公开视频接口，但在同一个 `try` 里还会继续做：

- `fetch_video_tags()`
- `enrich_video_info_with_html_hints()`

这意味着：

- 主信息其实可能已经拿到了
- 但只要补 `tags`、补 `tname` 的附加请求超时
- 整次调用仍然会被归类成 `public api` 失败

所以用户会看到“public api 超时”，但根因不一定是主信息接口失败，也可能只是补充步骤失败。

### 修复

#### 21.3 修复 `fetch_text()` 的压缩响应解码

- 新增 `decode_http_response_body()`
- 兼容 `gzip`
- 同时兼容 `deflate`
- 再按响应头里的 `charset` 或默认 `utf-8` 解码

修复后，HTML 兜底解析可以真正读到：

- `<title>`
- `og:title`
- `window.__INITIAL_STATE__`
- 页面里的 `videoData`

#### 21.4 把补充信息改成 best-effort

- `fetch_video_tags()` 失败时不再向上打断主流程
- 新增 `enrich_video_info_best_effort()`
- 公开视频接口和 `bilibili_api` 拿到主信息后，都只做“尽力补充”

修复后的策略是：

- 只要标题、封面、时长、UP、统计这些基础字段已经拿到
- 即使 `tags / keywords / HTML hints` 失败
- `/api/resolve-bili-link` 也应该继续返回可用的 `resolved` 结果

### 修复后的实际行为

对 `BV1dcX5BYESE` 复测后：

- `fetch_video_info_via_html()` 已能解析出标题
- `fetch_video_info()` 能稳定返回基础信息
- `/api/resolve-bili-link` 返回 `200`
- 前端预览可拿到：
  - `title`
  - `cover`
  - `duration`
  - `mid / up_name`
  - `stats`

### 21.5 补充：为什么只解析基础信息时又出现了 30 多秒

后续又出现过一个新现象：

- 用户只是把链接贴进输入框
- 前端只是请求 `/api/resolve-bili-link`
- 但整次预览解析花了三十几秒

排查后发现，慢的不是 HTML 解析本身，而是预览接口一度复用了完整解析链路。

在当时环境里，实测耗时接近这样：

- 公开视频接口超时约 `11s`
- `fetch_video_tags()` 再超时约 `10s`
- `bilibili_api` 再等待约 `20s`
- 真正能拿到基础信息的 HTML 解析只要不到 `1s`

也就是说：

- “只解析基础信息却要 30 多秒”不是因为基础信息难拿
- 而是因为接口在等待几个已经失败的慢接口超时

### 21.6 这次性能问题的修复

为了解决这个问题，解析链路做了拆分：

#### 预览快速链路

`/api/resolve-bili-link` 现在改成：

1. 先抽 `BV`
2. 优先走 HTML 快速解析
3. HTML 失败才退到公开视频接口
4. 再失败才退到 `bilibili_api`
5. 成功后直接构建 `resolved`

这条链路不再主动为了补 `tags / keywords / tname` 去等待慢接口。

## 26. 2026-04-05：智能问答流式输出时禁止自动滚底，并收紧底部留白

### 现象

- 智能问答逐字输出时，只要底部有新内容生成，滚动容器就会被强制拉到最底部
- 用户正在查看上方历史问答时，界面会反复跳走，无法稳定停留在当前阅读位置
- 问答区底部存在明显多余留白，视觉上离页面底边偏远

### 根因

- `web/static/app.js` 里的 `renderAssistant()` 在每次重绘后都执行了 `box.scrollTop = box.scrollHeight`
- 助手回复采用逐字流式渲染，意味着每个字符都会触发一次重绘，也就会不断触发一次“强制滚底”
- `web/static/style.css` 里给问答结果容器设置了过大的底部内边距，同时右侧助手面板高度预留过多，导致底部空白明显

### 修复

- 保留逐字流式输出，不改打字机效果
- 移除智能问答的自动滚底逻辑，改为在重绘后恢复用户当前的 `scrollTop`
- 用户在生成过程中可自由上下滚动，生成行为不再影响当前查看位置

## 27. 2026-04-05：智能问答继续提问时，上方推荐视频卡片跟随进度闪动

### 现象

- 智能问答里已经展示出来的推荐视频卡片，在用户继续提下一个问题时会反复闪动
- 闪动会随着“正在思考”的进度条刷新一起出现
- 视频标题、封面和卡片 hover 态看起来像是在不断重建

### 根因

- `web/static/app.js` 里的 `renderAssistant()` 之前会在两个场景下高频整段重绘：
  - 助手思考进度条刷新
  - 助手逐字输出回答
- 整个聊天 DOM 被反复替换后，历史消息里的推荐视频卡片和封面节点也会被一起重建
- 数据本身没有变化，闪动只是前端整段重绘造成的视觉抖动

### 修复

- 保留思考进度条和逐字输出效果
- 改成局部更新：
  - 进度刷新时只更新 pending 气泡
  - 打字机输出时只更新当前正在输出的那条消息内容
- 已经展示完成的历史推荐视频卡片不再跟着重建，封面保持稳定不闪

### 落点文件

- `web/static/app.js`
- `docs/07_问题排查与修复记录.md`

## 28. 2026-04-05：发送新消息后，聊天区需要自动定位到这条用户消息

### 现象

- 用户继续提下一个问题时，聊天区会停留在原来的阅读位置
- 如果上一轮对话较长，刚发出的新消息可能出现在视口更下方，不方便立刻把当前轮当作新的阅读起点

### 目标交互

- 每次发送新消息后，聊天区自动定位到这条用户消息
- 这条用户消息作为当前视口顶部附近的起点
- 历史对话仍然保留在上方，用户随时可以手动向上滚动查看
- 后续思考进度刷新和逐字输出不再改变这个定位结果

### 修复

- 在 `web/static/app.js` 里增加“发送后定位当前用户消息”的单独滚动函数
- 仅在用户消息刚插入聊天记录并完成渲染后执行一次
- 不恢复自动滚底，也不影响用户后续自由滚动

- `web/static/app.js`
- `docs/07_问题排查与修复记录.md`
