# 问题排查与修复记录

本文记录当前项目里已经确认过的一批真实问题和修复方式，方便后续排查时直接回看。

## 1. 运行时 LLM 配置缺失导致页面反复报错

### 现象

- 页面在 LLM 模式下分析失败
- 用户只看到连接错误，不知道去哪里补 URL / Key / provider / model

### 修复

- 增加运行时配置保存接口
- 页面内直接展示配置表单
- 失败后可自动拉起重配入口

## 2. B 站短链 / 跟踪链接 / av 链接解析不稳定

### 现象

- `b23.tv`
- `cm.bilibili.com`
- 旧版 `av` 链接

不能稳定进入统一解析链路。

### 修复

- 在解析入口统一展开和归一
- 提取 BV / av
- 构建统一 `resolved` 结果

## 3. 文案标题长期掉回提问句模板

### 现象

- 标题反复出现提问句
- 容易出现“为什么 / 怎么 / 哪种更容易起量”这类固定模板

### 修复

- 在 `CopywritingAgent` 和 `OptimizationAgent` 里增加标题清洗规则
- 对生活区 / 恋爱日常类脚本做更强约束

## 4. 前端文档和页面真实结构不一致

### 现象

- 文档长期写“双模块 + 聊天”
- 但当前页面实际已经是三模块

### 修复

- 统一改为：
  - 视频分析
  - 内容创作
  - 知识库管理
  - 右侧智能助手

## 5. 知识库检索把同一篇文档多个分片当多条结果显示

### 现象

- 搜索结果里会看到同一篇文档重复出现
- 有时甚至直接显示出后半段字段，例如 `"链接": "https://..."` 这种片段

### 根因

- Chroma 返回的是分片命中
- 展示层直接按分片逐条渲染，没有按 `document_id` 合并

### 修复

- 搜索结果渲染前先按文档去重
- 优先保留主片段内容

## 6. 知识库结果标题直接显示内部文档 ID

### 现象

- 标题会显示：
  - `分区热门榜:tech:BV...`
  - `全站热门榜:BV...`

### 根因

- 展示层回退逻辑直接把 `document_id` 当标题

### 修复

- 标题优先提取 `视频标题`
- 热门样本显示为“榜单名 + 分区 + 视频标题”
- 不再把原始 `BV...` 文档 ID 当主标题

## 7. 一级分类检索误召回

### 现象

- 点击 `舞蹈` 时，搜出“翻唱”“不知火舞”等不属于舞蹈分类的内容
- 甚至搜出 `分区热门榜:knowledge` 这类不相关样本

### 根因

- 旧逻辑只是拿分类词做纯向量检索
- 没有在召回后做分类过滤

### 修复

- 搜索先扩召回
- 再按一级分类规则做过滤
- 最后按文档去重

## 8. 热门样本同步只抓 5 个内部粗分区

### 现象

- 知识库里有 `分区热门榜:knowledge / tech / life / game / ent`
- 但用户在 `popular/rank` 页面看到的 `国创 / 舞蹈 / 音乐` 等真实分区没有单独同步

## 9. AI Key 过期后项目调不到模型

### 现象

- 页面切到 `LLM Agent 模式` 后直接报鉴权失败
- 聊天、视频分析、内容创作都退不出来结果
- 原因通常是旧 `LLM_API_KEY` 失效，或者 provider / base_url / model 已经换了

### 修复

- 项目实际读取的是根目录 `.env`
- 关键字段是：
  - `LLM_PROVIDER`
  - `LLM_API_KEY`
  - `OPENAI_API_KEY`
  - `LLM_BASE_URL`
  - `LLM_MODEL`
  - `LLM_REASONING_EFFORT`
  - `LLM_DISABLE_RESPONSE_STORAGE`
- 改完后重启 `python web/app.py`

### 备注

- 页面内运行时配置只适合临时改 `provider / url / key / model`
- 如果想把策略长期固定下来，还是改 `.env`
- 当前项目已支持把 `LLM_REASONING_EFFORT` 透传给模型调用

### 根因

- 旧实现只抓项目内部主分区

### 修复

- 改成抓真实一级分区榜
- 同步前清理旧的粗分区榜样本
- 给热门样本追加 `榜单链接`

## 9. 热门知识库同步全部失败：`invalid literal for int() with base 10: '0:47'`

### 现象

- 点击“自动更新热门知识库”后全部失败
- 报错里出现：
  - `invalid literal for int() with base 10: '0:47'`

### 根因

- 热门样本里部分 `duration` 不是纯数字，而是 `0:47` 这类字符串
- 旧逻辑直接 `int(duration)`，导致整条样本失败

### 修复

- 增加时长解析函数
- 兼容：
  - `0:47`
  - `01:23`
  - `1:02:03`
  - 数字秒数

## 10. 热门知识库同步全部失败：`title` 未定义

### 现象

- 同步更新后所有热门样本都失败

### 根因

- 给热门样本 metadata 增加 `title` 字段时，直接引用了未定义变量
- 这个错误属于运行时错误，不会在简单编译检查时暴露

### 修复

- 在 `_ingest_hot_items()` 内先显式提取标题
- 再写入 metadata

## 11. 当前环境缺少 Chroma 依赖导致知识库不可写

### 现象

- 点击知识库上传、查看、检索、同步时直接失败
- 或状态里显示 `Chroma backend not initialized`

### 根因

- 当前运行环境没装：
  - `chromadb`
  - `langchain-chroma`
- 旧实现遇到这种情况时，知识库会整体不可用

### 修复

- 知识库后端改成：
  - 优先使用 Chroma
  - 缺依赖时自动回退到本地 JSON 存储
- 状态区额外显示当前后端类型和回退说明

## 12. JSON fallback 文件里为什么会看到向量数据

### 现象

- 打开 `vector_db/bilibili_knowledge__fallback_store.json`
- 发现每条记录里除了正文，还有一大段 `embedding`
- 容易误以为“系统把整份向量 JSON 直接发给了大模型”

### 根因

- 这份文件是知识库 fallback 存储，不是单独导出的“原始热门样本 JSON”
- 每条记录当前同时保存：
  - `text`
  - `metadata`
  - `embedding`
- 其中：
  - `text` 才是实际可读的结构化业务内容
  - `metadata` 是过滤和展示辅助字段
  - `embedding` 只用于检索排序

### 修复 / 当前真实行为

- 当前 Agent 或知识库接口不会把整份 fallback 文件直接发给大模型
- 当前检索链路只会把命中的：
  - `id`
  - `text`
  - `metadata`
  - `score`
  返回给上层
- 大模型真正看到的是命中结果，不是整库，也不是整段向量数组
- 页面里的“关键词检索”接口本身也不会调用大模型，只是后端检索后返回前端展示

## 13. 当前排查热门知识库问题时的推荐顺序

出现热门样本问题时，建议按下面顺序查：

1. 先看后端有没有重启
2. 再看知识库状态里当前后端是不是 `langchain_chroma`、`chromadb` 或 `json_fallback`
3. 再看同步是不是按新逻辑重新执行过
4. 看 `knowledge_sync.py` 里的热门源和分区列表
5. 看 `web/app.py` 里的知识库搜索过滤逻辑
6. 看 `web/static/app.js` 的展示去重和标题格式化逻辑
