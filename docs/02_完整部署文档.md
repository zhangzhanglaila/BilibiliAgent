# 完整部署文档

## 1. 环境要求

- Python 3.10 及以上
- Windows / macOS / Linux
- 能访问 B 站公开页面和公开接口
- 如果要启用 LLM，需要能访问你配置的模型服务地址

## 2. 安装依赖

在项目根目录执行：

```bash
pip install -r requirements.txt
```

当前依赖见：

- `langgraph`
- `langchain`
- `langchain-openai`
- `bilibili-api-python`
- `python-dotenv`
- `requests`
- `flask`

## 3. 初始化 `.env`

先复制模板：

### Windows

```bash
copy .env.example .env
```

### macOS / Linux

```bash
cp .env.example .env
```

然后编辑项目根目录的 `.env`。

## 4. 环境变量说明

下面这些变量都来自当前代码里的 `config.py`，文档需要和代码保持一致。

### 4.1 LLM 相关

- `LLM_PROVIDER`
  默认值 `openai`
- `LLM_API_KEY`
  为空时，Web 主流程进入规则模式
- `LLM_BASE_URL`
  默认值 `https://zapi.aicc0.com/v1`
- `LLM_MODEL`
  默认值 `gpt-5.4`
- `LLM_TIMEOUT_SECONDS`
  默认值 `75`
- `LLM_MAX_RETRIES`
  默认值 `2`
- `LLM_RETRY_BACKOFF_SECONDS`
  默认值 `1.6`

### 4.2 B 站相关

- `BILI_SESSDATA`
  可选，默认空
- `BILI_BILI_JCT`
  可选，默认空

### 4.3 项目运行相关

- `DB_PATH`
  默认值 `bilibili_agents.db`
- `REQUEST_INTERVAL`
  默认值 `1.2`
- `DEFAULT_PARTITION`
  默认值 `knowledge`
- `DEFAULT_PEER_UPS`
  默认值 `546195,15263701,777536`

## 5. 推荐配置示例

下面是一个可直接参考的示例，真实 key 不要写进仓库：

```env
BILI_SESSDATA=
BILI_BILI_JCT=

LLM_PROVIDER=openai
LLM_API_KEY=your-real-key
LLM_BASE_URL=https://zapi.aicc0.com/v1
LLM_MODEL=gpt-5.4
LLM_TIMEOUT_SECONDS=75
LLM_MAX_RETRIES=2
LLM_RETRY_BACKOFF_SECONDS=1.6

DB_PATH=bilibili_agents.db
REQUEST_INTERVAL=1.2
DEFAULT_PARTITION=knowledge
DEFAULT_PEER_UPS=546195,15263701,777536
```

如果你当前 provider 对复杂请求响应慢，可以只在本地 `.env` 覆盖：

```env
LLM_MODEL=gpt-4.1-mini
```

前提是你的 provider 支持这个模型名。运行时以 `.env` 为准，不以 `.env.example` 或 `config.py` 默认值为准。

## 6. 启动方式

```bash

```



## 7. 当前 Web 页面包含什么

页面由 3 个区域组成：

- 顶部状态区
- 左侧业务工作区
- 右侧智能对话助手

左侧业务工作区通过顶部标签切换两个模块：

- `视频分析`
- `内容创作`

不是旧文档里的“悬停伸缩双面板”。

## 8. 当前 Web 主接口

### 8.1 页面基础接口

- `GET /api/runtime-info`
  返回运行模式、是否启用聊天、token 提示、切换提示
- `POST /api/resolve-bili-link`
  输入视频链接，只做解析，不做分析

### 8.2 内容创作主接口

- `POST /api/module-create`

请求体示例：

```json
{
  "field": "AI效率",
  "direction": "教程",
  "idea": "想做一条关于 AI 自动剪辑提高效率的实战视频",
  "partition": "tech",
  "style": "干货"
}
```

返回内容核心字段：

- `normalized_profile`
- `seed_topic`
- `partition`
- `style`
- `chosen_topic`
- `topic_result`
- `copy_result`

### 8.3 视频分析主接口

- `POST /api/module-analyze`

请求体示例：

```json
{
  "url": "https://www.bilibili.com/video/BV1xx411c7mD"
}
```

如果前端已经解析过视频，也会附带：

```json
{
  "url": "https://www.bilibili.com/video/BV1xx411c7mD",
  "resolved": {}
}
```

返回内容核心字段：

- `resolved`
- `performance`
- `topic_result`
- `optimize_result`
- `analysis`
- `reference_videos`
- `copy_result`

### 8.4 智能对话接口

- `POST /api/chat`

请求体示例：

```json
{
  "message": "帮我分析这个视频为什么没起量",
  "history": [],
  "context": {
    "field": "AI效率",
    "direction": "教程",
    "idea": "做 AI 自动剪辑",
    "partition": "tech",
    "style": "干货",
    "videoLink": "https://www.bilibili.com/video/BV1xx411c7mD"
  }
}
```

返回内容核心字段：

- `reply`
- `suggested_next_actions`
- `reference_links`

## 9. 兼容接口

虽然主界面现在只强调 2 个模块和 1 个助手，但这批接口仍然保留：

- `POST /api/topic`
- `POST /api/copy`
- `POST /api/operate`
- `POST /api/optimize`
- `POST /api/pipeline`

### 9.1 兼容接口的 LLM 行为

这里要特别注意，旧文档通常容易漏掉：

- `/api/topic`
  不直接调用 LLM
- `/api/copy`
  有 Key 时会调用 `LLMClient`
- `/api/operate`
  有 Key 时生成评论回复会调用 `LLMClient`
- `/api/optimize`
  有 Key 时会调用 `LLMClient`
- `/api/pipeline`
  因为串联了 `copy / operate / optimize`，有 Key 时可能消耗 token

## 10. 当前运行模式

### 10.1 规则模式

条件：

- `LLM_API_KEY` 为空

结果：

- `/api/module-create` 走规则链路
- `/api/module-analyze` 走规则链路
- `/api/chat` 不可用

### 10.2 LLM Agent 模式

条件：

- `LLM_API_KEY` 有值

结果：

- `/api/module-create` 走 `LLMWorkspaceAgent`
- `/api/module-analyze` 走 `LLMWorkspaceAgent`
- `/api/chat` 可用

### 10.3 LLM 回退逻辑

当前代码里实际存在两层回退：

1. 先尝试 `LLMWorkspaceAgent`
2. 如果满足条件，再尝试单次 direct LLM fallback

如果 fallback 成功，结果里会带：

- `llm_warning`

前端会把这个 warning 提示出来。

## 11. 当前前端行为

### 11.1 视频分析模块

- 粘贴链接后会自动解析
- 自动请求 `/api/resolve-bili-link`
- 预览区先显示视频信息
- 点击“一键解析并分析视频”后再调用 `/api/module-analyze`

### 11.2 内容创作模块

- 输入领域、方向、想法、分区、风格
- 点击按钮后调用 `/api/module-create`
- 结果区可一键复制标题、脚本、简介、置顶评论

### 11.3 智能对话助手

- 有快捷提问按钮
- 有浏览器语音输入支持
- 有打字机式回复效果
- 可展示参考视频链接

## 12. 数据库说明

项目默认使用 SQLite。

默认数据库文件：

```text
bilibili_agents.db
```

当前表用途：

- `video_metrics`
  保存视频指标历史

主要由：

- `OptimizationAgent`
- `SQLiteStore`

负责读写。

## 13. 常见故障

### 13.1 链接能提取 BV，但视频解析失败

可能原因：

- B 站公开接口对当前视频返回异常
- 当前网络环境无法访问 B 站页面
- 视频不是公开视频

### 13.2 有 Key 但助手还是灰的

优先检查：

- `.env` 中 `LLM_API_KEY` 是否真的有值
- 服务是否已经重启
- `/api/runtime-info` 返回的 `chat_available` 是否为 `true`

### 13.3 有 Key 但分析很慢或者报超时

优先检查：

- `LLM_BASE_URL` 是否可用
- `LLM_MODEL` 是否是 provider 支持的模型名
- `LLM_TIMEOUT_SECONDS`
- `LLM_MAX_RETRIES`

如果当前 provider 对复杂分析链路响应慢，可以先尝试：

- 换更快的模型
- 降低重试次数
- 适当调大超时

### 13.4 为什么没有 Key 也能跑

因为当前项目保留了完整的规则链路：

- 选题规则
- 视频表现判断规则
- 文案 fallback
- 优化建议 fallback

所以没有 Key 也能使用主流程，只是聊天助手不可用。
