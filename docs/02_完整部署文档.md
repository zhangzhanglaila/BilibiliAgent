# 完整部署文档

## 1. 环境要求

- Python 3.10 及以上
- Windows / macOS / Linux
- 能访问 B 站公开接口
- 如果启用 LLM，需要能访问你配置的模型服务地址

推荐在独立虚拟环境中安装。

## 2. 安装依赖

项目根目录执行：

```bash
pip install -r requirements.txt
```

当前依赖以 [requirements.txt](../requirements.txt) 为准，核心包括：

- `flask`
- `bilibili-api-python`
- `chromadb`
- `langchain-chroma`
- `langgraph`
- `sentence-transformers`
- `python-docx`
- `pypdf`

## 3. 初始化 `.env`

复制模板：

### Windows

```bash
copy .env.example .env
```

### macOS / Linux

```bash
cp .env.example .env
```

模板文件见：[.env.example](../.env.example)

## 5.1 下次自己替换 AI Key 的最短流程

只改项目根目录的 `.env`：

1. 替换 `LLM_API_KEY`
2. 同步替换 `OPENAI_API_KEY`
3. 如供应商或网关有变化，再替换 `LLM_PROVIDER`
4. 如接口地址有变化，再替换 `LLM_BASE_URL`
5. 如模型有变化，再替换 `LLM_MODEL`
6. 如要调低推理成本或速度，再改 `LLM_REASONING_EFFORT`
7. 重启 `python web/app.py`

补充说明：

- 页面运行时表单适合临时改 `provider / url / key / model`
- `.env` 适合保存长期配置
- `LLM_REASONING_EFFORT`、`LLM_DISABLE_RESPONSE_STORAGE` 当前以 `.env` 为准

## 8. 当前知识库部署注意事项

### 8.1 知识库数据目录

知识库默认写到：

```text
./vector_db
```

如果需要迁移目录，改 `VECTOR_DB_PATH` 即可。

当前项目知识库后端优先使用 Chroma。

如果运行环境里缺少 `chromadb` 或 `langchain-chroma`：

- 系统会自动回退到 `vector_db` 目录下的本地 JSON 存储
- 上传、同步、查看、检索仍可继续使用
- 但如果你希望恢复 Chroma，就需要把 `requirements.txt` 里的相关依赖装齐

### 8.2 Embedding 模型

项目优先使用 `sentence-transformers`。

如果本地模型不可用：

- 不会导致知识库彻底不可用
- 会自动回退到 deterministic embedding
- 只是召回效果会比正常 embedding 弱

### 8.3 文件导入格式

当前支持：

- `.txt`
- `.md`
- `.docx`
- `.pdf`

## 9. `module-analyze` 结构扩展与性能优化记录

这一节记录 Web 端 `web.run_llm_module_analyze` 最近一次比较关键的排查结果，后面如果你继续调这个接口，可以先看这里。

### 9.1 遇到的问题

问题 1：返回内容太薄

- 原来 `analysis` 只要求输出：
  - `analysis_points`
  - `followup_topics / next_topics`
  - `title_suggestions`
  - `cover_suggestion`
  - `content_suggestions`
- 所以模型虽然能判断“为什么容易火 / 为什么偏弱”，但结构层面的对标、脚本、封面、标签、发布时间这些信息都没有被强制返回。

问题 2：接口总耗时很高

- LangSmith 里可以看到：
  - `knowledge_base.retrieve()` 很快
  - 核心 LLM 调用本身也不是最主要瓶颈
  - 真正拖慢的是外层同步阻塞和重复处理
- 当时的慢点主要有 4 个：
  1. 同一次分析里 `market_snapshot` 构建了两次
  2. 参考视频区块会再串行走一轮搜索、相关推荐、详情补齐
  3. `LLMWorkspaceAgent.run_structured()` 末尾还会做额外 reflection
  4. 历史检索 / 长期记忆写入也会占主链路时间

### 9.2 这次做了什么

#### 输出结构扩展

保持原有字段不删不改，只在 `analysis` 下新增这些字段：

- `benchmark_analysis`
- `remake_script_structure`
- `advanced_title_sets`
- `cover_plan`
- `tag_strategy`
- `publish_strategy`
- `reusable_hit_points`

同时为了不改前端渲染逻辑，还把新增信息同步回旧字段：

- `analysis.title_suggestions`
- `analysis.cover_suggestion`
- `analysis.content_suggestions`
- `optimize_result.optimized_titles`
- `optimize_result.cover_suggestion`
- `optimize_result.content_suggestions`

这样前端即使完全不改，也会比以前展示得更丰富。

#### 性能优化

这次只动“外层冗余”，不改核心链路：

- 核心链路仍然是：
  - 接收视频数据
  - RAG 检索
  - LLM 分析
  - 结构化 JSON 输出
  - 返回前端

具体优化点：

1. `api_module_analyze()` 里只构建一次 `market_snapshot`，fallback 复用同一份数据。
2. `build_market_snapshot()` 改成并行抓取：
   - `hot_board`
   - `partition_samples`
   - `peer_samples`
3. `run_llm_module_analyze()` 改成只允许 `retrieval` 工具，避免模型在分析链路里乱走 `web_search / code_interpreter / hot_board_snapshot`。
4. `module_analyze` 调 `run_structured()` 时关闭：
   - 历史上下文检索
   - reflection 二次质检
   - 同步长期记忆写入
5. 参考视频区块不再优先走外部二次抓取，而是先复用：
   - 本次 RAG 命中的知识库高表现视频
   - 已经拿到的 `market_snapshot`
6. 新增 `finalize_module_analyze_result()`，统一做：
   - 结构补齐
   - 字段兼容
   - 参考视频拼装
   - 低表现文案兜底

### 9.3 现在要重点看哪些文件

- [web/app.py](../web/app.py)
  - `build_market_snapshot()`
  - `build_reference_videos_from_market_snapshot()`
  - `build_module_analyze_reference_videos()`
  - `normalize_module_analysis_payload()`
  - `finalize_module_analyze_result()`
  - `run_llm_module_analyze()`
  - `run_llm_module_analyze_fallback()`
  - `api_module_analyze()`
- [agents/llm_workspace_agent.py](../agents/llm_workspace_agent.py)
  - `run_structured()`
  - `_save_memory_async()`

### 9.4 后面如果还想继续压时间，优先看这几个方向

1. 先看 LangSmith 里 `web.run_llm_module_analyze` 的子节点，确认现在主耗时还在不在 LLM 本身。
2. 如果仍然慢，先检查当前模型服务真实单次响应时间，不要误把上游模型延迟当成业务代码问题。
3. 如果要继续压缩，优先做：
   - 缩短 `response_contract`
   - 进一步压缩 `parsed_video / market_snapshot` 提示词体积
   - 针对 `module_analyze` 单独使用更快的模型配置
4. 不建议再把“参考视频展示”改回大量同步搜索和详情补齐，那会很容易把总耗时重新拉高。

## 9. 生产部署建议

- 把 `.env` 放到服务器本地，不要进仓库
- `VECTOR_DB_PATH` 放到独立持久化目录
- `model_cache` 单独持久化，避免重复下载 embedding 模型
- 使用反向代理托管 Flask 服务
- 如果要启用 LLM，优先先在页面里验证运行时配置，再切正式流量
