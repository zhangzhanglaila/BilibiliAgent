# 问题排查与修复记录

本文记录这轮开发中实际遇到过的问题、定位过程，以及最终采取的修复方式，方便后续继续维护时快速回溯。

## 1. LLM Agent / fallback 频繁报 `Connection error`

### 现象

- 页面在 `LLM Agent 模式` 下分析失败，提示：
  - `LLM Agent 分析失败：Connection error.`
  - `LLM fallback 也失败：Connection error.`
- 用户不知道应该去哪里补 `base_url / api_key / provider / model`。

### 原因

- 运行时没有拿到可用的 LLM 配置。
- 失败后前端只报错，没有把重配入口直接暴露出来。

### 解决过程

1. 增加运行时 LLM 配置保存逻辑，支持在页面内直接填写 `URL / Key / 模型供应商 / 模型名`。
2. 当 LLM 请求失败且属于可重配场景时，后端返回 `show_runtime_config` 和 `runtime_payload`。
3. 前端收到这类错误后：
   - 在运行模式区域下方自动展开配置表单；
   - 触发震动 / 抖动提示；
   - 允许用户直接补齐配置后重试当前动作。

### 当前结果

- 用户不需要再改 `.env` 才能继续使用。
- 报错后会直接引导到可填写的运行时配置区域。

## 2. B 站跟踪链接 / `av` 链接解析失败

### 现象

- 类似 `cm.bilibili.com/...` 的推广跟踪链接无法正常解析视频。
- 类似 `https://www.bilibili.com/video/av1105942827/...` 的旧版 `av` 链接也会失败。

### 原因

- 早期逻辑只稳定支持标准 `BV` 链接。
- 对带追踪参数的链接没有先做规范化处理。

### 解决过程

1. 增加跟踪链接宿主识别。
2. 从 `cm.bilibili.com` 这类 URL 的参数中提取真实视频地址。
3. 为旧版 `av` 号链接补充解析逻辑，统一转换到内部可处理的视频标识。

### 当前结果

- 常见分享链接、追踪链接、`av` 链接都能进入同一条解析链路。

## 3. 标题优化和后续题材长期落入固定模板

### 现象

- 输出中反复出现：
  - `做哪种切口更容易起量`
  - `哪种表达更容易被点进来`
  - `下一条最适合拍什么`
- 标题和选题带明显提问感、教学感，不像生活区 vlog。

### 原因

- 规则链路和 LLM 提示里都保留了旧模板思路。
- 对“叙事型、陈述型、生活化标题”的约束不够强。

### 解决过程

1. 移除固定三连反问式模板。
2. 重写标题清洗和规范化逻辑，禁止：
   - 提问句；
   - `90 秒讲清`
   - `做对这一步`
   - `为什么更容易起量`
   - `更容易被推荐`
3. 在 LLM 与 fallback 提示里明确要求：
   - 标题必须是陈述型、叙事型；
   - 风格贴近 B 站生活区 vlog；
   - 不得把原题后面机械拼接模板尾巴。

### 当前结果

- 标题更接近日常记录、见面流程、轻松口语化表达。

## 4. 异地恋 / 情侣日常脚本生成成了运营话术

### 现象

- 口播脚本出现：
  - `先拿一个最容易验证的切口`
  - `测反馈`
  - `完播`
  - `方向跑偏`
  - `实战拆解`
- 内容缺场景、缺情绪、缺画面，无法直接对镜念。

### 原因

- 脚本生成提示把“内容创作”误导成“运营拆解”。
- 没有针对情侣日常 / 520 vlog 单独加风格约束。

### 解决过程

1. 在视频分析与 fallback 提示中增加明确限制：
   - 禁止输出运营黑话；
   - 必须贴合酒店、早午餐、逛街拍照、小清吧、异地恋见面等场景。
2. 保留固定时长结构：
   - `0-8s 开头钩子`
   - `8-28s 核心画面 1`
   - `28-56s 核心画面 2`
   - `56-75s 结尾互动`
3. 对结果做归一化和兜底，避免再次落回运营文案。

### 当前结果

- 这类题材输出已经改成更像情侣日常短视频口播的脚本。

## 5. `BV1vSPCzkEEg` 被误判成“知识区”

### 现象

- 用户提供的视频实际是偏 `颜值 / 卡点 / 身材展示` 的内容。
- 系统却显示成 `知识`，后续分析和参考视频都开始跑偏。

### 原因

- B 站公开视频接口对这条视频返回了：
  - `tid=255`
  - `tname` 为空
- 旧逻辑在 `tname` 为空时，会默认回落到 `knowledge`。

### 解决过程

1. 扩充分区映射：
   - 根据 `title + keywords + tags + tname` 综合判断；
   - 将 `舞蹈 / 卡点 / 变速卡点 / 颜值 / 美女 / 身材` 统一映射到 `ent`。
2. 补充 `tid` 提示：
   - 对 `tid=255` 增加 `颜值·网红舞` 的兜底识别。
3. 在主题推断中新增更细的语义：
   - `颜值卡点展示`
   - `变速卡点舞蹈展示`

### 当前结果

- 这条视频现在会落到：
  - `partition=ent`
  - `partition_label=娱乐`
  - `tname=颜值·网红舞`
  - `topic=颜值卡点展示`

## 6. 参考视频只跟标题像，内容却不相关

### 现象

- “同赛道高表现参考视频”经常只是标题撞词，实际内容不在同一赛道。
- 用户看到的是“标题差不多，但内容完全不相干”的结果。

### 原因

- 早期排序几乎只依赖标题文本匹配。
- 搜索候选没有补齐自己的 `tname / keywords / topic / partition`。
- 当前视频自己的关键词还曾被错误序列化成 `['变速卡点']` 这种脏字符串，导致查询词也被污染。

### 解决过程

1. 修复关键词提取：
   - `extract_video_keywords()` 改为支持 `str / list / tuple / set / dict`；
   - 不再把列表直接转成字符串；
   - 清掉 `['xx']` 这种序列化污染。
2. 参考视频排序从“只看标题”改成“先补语义再排”：
   - 给候选视频补齐 `tid / tname / keywords / partition / topic / style`；
   - 用这些语义字段参与相关性判断。
3. 在严格相关模式下增加约束：
   - 目标关键词必须命中；
   - 分区尽量一致；
   - 对当前视频相关推荐和粗排靠前候选优先做二次补齐。

### 当前结果

- 参考视频不再只是标题相似，而是会优先回到同一内容语义轨道。

## 7. 旧缓存导致修复后的结果前端还是显示旧值

### 现象

- 即使后端逻辑已经修好，前端仍可能继续显示旧的 `resolved` 结果，例如：
  - 还显示 `知识区`
  - 继续沿用旧的 topic / style

### 原因

- 前端会缓存一次解析后的 `resolved`。
- 后端之前只校验 `BV / title / stats`，没有校验这份缓存是否属于新版本结构。

### 解决过程

1. 在 `resolved` 结构里加入 `resolved_version`。
2. 后端复用缓存前，必须检查：
   - `resolved_version`
   - `partition`
   - `topic`
   - `style`
3. 旧缓存自动失效，强制重新解析。

### 当前结果

- 老结果不会继续污染新逻辑。

## 8. 参考视频相关性修复后，接口耗时变长

### 现象

- 语义补齐做得太激进时，参考视频接口会明显变慢。

### 原因

- 给过多候选视频都补了二次详情，导致请求数膨胀。
- 同时还打了过多搜索词。

### 解决过程

1. 改成两阶段：
   - 先粗排；
   - 再只给前排候选补齐语义。
2. 语义搜索词收敛到更少的几组。
3. 搜索结果数量在严格相关模式下做缩减。
4. 参考视频详情优先用公开视频接口和 tag，只有必要时才回 HTML。

### 当前结果

- 相关性提升后，耗时也被压回可接受范围内。

## 9. 这次排查沉淀下来的经验

### 核心结论

1. B 站公开视频接口并不总能给出完整 `tname`，不能直接把空值当未知再默认回 `knowledge`。
2. 参考视频相关性不能只靠标题，必须尽量引入候选自身的语义字段。
3. 前端缓存如果没有版本控制，会让“代码已修复但页面没变”的问题长期存在。
4. 和 LLM 相关的报错不能只提示失败，必须把“下一步怎么配置”直接暴露给用户。
5. 生活区 / 恋爱日常 / vlog 的文案生成，必须在提示里明确禁止运营术语，否则很容易跑偏。

### 建议后续继续保持

- 任何新的解析字段都同步进入 `resolved_version` 管理。
- 任何新的“同赛道推荐”逻辑都优先校验候选自身的语义字段。
- 任何会被用户直接感知的失败，都提供可操作的恢复入口，而不是只报错。
