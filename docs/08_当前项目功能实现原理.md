## 2. 页面层怎么工作

### 2.1 页面结构

当前页面是三模块工作台：

- 视频分析
- 内容创作
- 知识库管理
- 右侧智能助手

### 2.2 首屏初始化

页面首屏会拿服务端下发的：

- `window.__INITIAL_RUNTIME__`

前端用它初始化：

- 运行模式
- 开关状态
- 聊天可用性
- 已保存配置摘要

### 2.3 状态管理

前端核心状态集中在 `state`：

- 当前激活模块
- 当前运行模式
- 已解析视频信息
- 知识库子 Tab
- 知识库表单
- 知识库结果缓存
- 聊天记录

## 3. Web 接口层怎么分流

[web/app.py](../web/app.py) 当前承担三件事：

1. 接收前端请求
2. 依据模式切规则链路或 LLM 链路
3. 统一返回前端需要的 JSON

## 4. 视频分析链路原理

### 4.1 入口

- `/api/resolve-bili-link`
- `/api/module-analyze`

### 4.2 过程

1. 前端先提交视频链接
2. 后端提取 BV / av / 跟踪链接里的真实视频标识
3. 拉取公开视频信息
4. 构造统一的 `resolved` 结构
5. 根据当前模式进入不同分支

### 4.3 规则模式

规则模式下主要做：

- 自动分类视频分区
- 计算表现分数
- 判断热门爆款 / 播放偏低
- 生成规则分析
- 低表现时生成规则优化与新文案
- 按主题和视频信息挑选参考视频

### 4.4 LLM Agent 模式

LLM 模式下交给 `LLMWorkspaceAgent`：

- 后端仍然先做真实数据解析
- 再把结构化结果作为工具上下文交给 LLM
- 如果 Agent 中枢失败，部分场景会降级到单次 LLM fallback

## 5. 内容创作链路原理

### 5.1 入口

- `/api/module-create`

### 5.2 规则模式

规则模式主要做：

1. 根据 `field / direction / idea` 生成 `seed_topic`
2. 用 `TopicAgent` 出选题
3. 选主选题
4. 用规则模式专用 `CopywritingAgent` 生成文案

### 5.3 LLM Agent 模式

LLM 模式下：

- 把创作输入整理成 briefing
- 交给 `LLMWorkspaceAgent`
- 由 Agent 决定如何调工具和组织结果

## 6. 知识库链路原理

知识库是当前项目变化最大的部分之一。

### 6.1 向量后端

当前知识库后端优先使用 Chroma。

如果运行环境里缺少 `chromadb` 或 `langchain-chroma`，会自动回退到本地 JSON 存储。

实现文件：

- [knowledge_base.py](../knowledge_base.py)

主要能力：

- 文档切片
- 向量写入
- 向量检索
- 文档采样
- 元数据过滤
- 缺依赖时的本地 fallback 存储

当前要特别区分两层：

1. 存储层
   - Chroma 模式下，文本、metadata、向量存在 Chroma 自己的数据文件里
   - JSON fallback 模式下，文本、metadata、向量直接写进 `vector_db/bilibili_knowledge__fallback_store.json`
2. 调用层
   - 上层接口仍然统一走 `KnowledgeBase.retrieve()`
   - 因此不管底层是 Chroma 还是 JSON fallback，调用方拿到的返回结构保持一致

### 6.2 Embedding

优先使用：

- `sentence-transformers`

如果本地不可用：

- 自动回退 deterministic embedding

### 6.3 文件导入

入口：

- `/api/knowledge/upload`
- `main.py ingest-file`

支持：

- txt
- md
- docx
- pdf

导入时会：

1. 读取原文
2. 归一化文本
3. 生成内容哈希
4. 去重
5. 切片写入当前知识库后端（优先 Chroma，否则 JSON fallback）

### 6.4 热门样本同步

入口：

- `/api/knowledge/update`
- `main.py update-kb`

当前同步源包括：

- 全站热门榜
- 每周必看
- 入站必刷
- 真实一级分区热门榜

当前真实一级分区榜包括：

- 番剧
- 国创
- 纪录片
- 动画
- 游戏
- 鬼畜
- 音乐
- 舞蹈
- 知识
- 科技
- 汽车
- 运动
- 动物圈
- 生活
- 娱乐
- 影视
- 电影
- 电视剧
- 时尚

当前同步时会额外保存：

- `board_type`
- `bvid`
- `title`
- `board_url`
- `partition`

同时结构化正文里也会写入：

- 榜单来源
- 视频标题
- 分区
- BVID
- 榜单链接
- 视频链接

也就是说，知识库里真正可读的业务内容主要在正文 `text` 里。
`metadata` 是辅助过滤和展示字段，`embedding` 只用于召回，不作为展示正文。

### 6.5 热门样本展示

前端展示层当前做了两类修正：

1. 同一文档多个分片会先合并
2. 卡片标题优先显示榜单名、分区和视频标题

因此不会再直接把：

- `分区热门榜:tech:BV...`

这种内部文档 ID 当标题。

### 6.6 一级分类检索

知识库检索当前不是“直接拿下拉值做纯向量检索”。

当前实现是：

1. 先扩召回
2. 如果下拉值属于一级分类
3. 再按分类规则过滤
4. 最后按 `document_id` 去重

这样能减少：

- 仅标题里碰巧出现关键词的误召回
- 不同分片重复展示

## 7. 运行模式原理

当前运行模式分成两层：

### 7.1 默认状态

由 `.env` 决定。

### 7.2 页面运行时状态

由页面上的运行模式开关控制。

当前关键点：

- 页面可以保存运行时 LLM 配置
- 保存后不需要改 `.env`
- 切模式只影响 Web 工作台

## 8. LLM Agent 原理

`LLMWorkspaceAgent` 当前是 Web LLM 模式的统一入口。

主要工具包括：

- `retrieval`
- `creator_briefing`
- `video_briefing`
- `hot_board_snapshot`
- `web_search`
- `code_interpreter`

### 8.1 Retrieval 工具到底把什么给模型

这里容易误解，当前不是把整个知识库文件直接发给大模型。

实际链路是：

1. Agent 调 `retrieval`
2. `retrieval` 内部调用 `KnowledgeBase.retrieve()`
3. 只取 top-k 命中结果
4. 把每条命中的：
   - `id`
   - `text`
   - `metadata`
   - `score`
   作为工具 observation 放进 Agent scratchpad
5. 大模型只看到这些命中结果，不会直接看到整库，也不会直接消费向量数组

所以：

- `embedding` 的作用是“参与检索排序”
- 真正进模型上下文的是“命中文本 + metadata + score”
- 页面里的“关键词检索”接口本身也不会调用大模型，只是后端检索后返回前端展示

典型行为：

- 根据任务目标决定调用哪些工具
- 汇总工具结果
- 按约定结构返回给前端

## 9. CLI 原理

[main.py](../main.py) 当前提供 7 个命令：

- `topic`
- `copy`
- `operate`
- `optimize`
- `pipeline`
- `ingest-file`
- `update-kb`

CLI 主要用于：

- 调试 Agent
- 导入知识文件
- 手动同步知识库

## 10. 当前版本的真实边界

### 10.1 页面模式不会覆盖 CLI

这两个体系是分开的。

### 10.2 知识库不依赖 LLM 才能工作

上传、同步、查看、检索都能在无 Key 下运行。

### 10.3 热门样本同步是“结构化样本沉淀”

它不是简单存原始视频 JSON，而是把视频标题、分区、互动点、评论热词、榜单来源等整理后入库。
