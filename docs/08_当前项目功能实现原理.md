# 当前项目功能实现原理

这份文档不是“怎么用”，而是说明当前项目里每个功能到底是怎么实现的，背后的判断逻辑是什么，代码之间是怎么串起来的。

---

## 1. 项目整体是怎么工作的

当前项目可以分成 5 层：

1. `web/templates/index.html` + `web/static/app.js`
   - 前端页面、表单、结果渲染、按钮事件、进度条、聊天面板都在这一层。
2. `web/app.py`
   - Flask Web 入口。
   - 负责接收前端请求、解析链接、组装上下文、切换运行模式、返回 JSON。
3. `agents/*.py`
   - 各个业务 Agent 的核心逻辑。
   - 包括选题、文案、运营、优化，以及有 Key 时使用的 `LLMWorkspaceAgent`。
4. `graph.py` + `main.py`
   - 把多个 Agent 串成一条完整流水线，供 CLI 或接口复用。
5. `config.py` + `llm_client.py` + `db.py` + `models.py`
   - 配置读取、LLM 调用、SQLite 持久化、数据结构定义。

项目的核心思路不是“所有事情都让大模型自由发挥”，而是：

- 无 Key 时，走规则链路，尽量靠公开数据、启发式评分和模板化结构完成任务。
- 有 Key 时，切到严格受控的 `LLMWorkspaceAgent`，让 LLM 基于工具返回的数据做分析和生成。

---

## 2. 两种运行模式是怎么切换的

当前版本的运行模式，不再只是“前端看 `.env` 写死显示什么”，而是分成两层：

1. 服务刚启动、页面第一次打开时
   - 先按 `.env` 判断默认状态
2. 页面已经打开之后
   - 再由运行模式区域右侧的开关控制当前这次运行时到底走哪种模式

也就是说：

- `.env` 决定“默认进来时是什么状态”
- 页面开关决定“当前运行时要不要启用 LLM Agent”

### 2.1 首次进入系统时，默认状态怎么定

这一层入口在：

- `config.py` 的 `AppConfig.llm_enabled()`
- `web/app.py` 顶部的：
  - `RUNTIME_LLM_ENABLED = CONFIG.llm_enabled()`
  - `RUNTIME_LLM_CONFIG = ... if CONFIG.llm_enabled() else None`
- `web/app.py` 的 `build_runtime_payload()`
- `web/app.py` 的 `index()`

工作方式是：

- 如果 `.env` 里有 `LLM_API_KEY`
  - `RUNTIME_LLM_ENABLED` 初始为 `True`
  - 页面首屏直接渲染成“LLM Agent 模式，开关开启”
- 如果 `.env` 里没有 `LLM_API_KEY`
  - `RUNTIME_LLM_ENABLED` 初始为 `False`
  - 页面首屏直接渲染成“无 Key 逻辑模式，开关关闭”

这一点非常重要，因为当前页面不是先写死成关闭，再靠 JS 异步改状态，而是：

- 后端 `index()` 先把 `build_runtime_payload()` 结果传给模板
- `web/templates/index.html` 用 `initial_runtime` 直接渲染首屏
- `web/static/app.js` 再用 `window.__INITIAL_RUNTIME__` 初始化前端状态

所以用户一进页面时，看到的开关状态就应该已经和后端当前默认模式一致。

### 2.2 页面里的开关怎么切模式

这一层入口在：

- 前端：`web/static/app.js`
  - `toggleRuntimeMode()`
  - `submitRuntimeConfig()`
- 后端：`web/app.py`
  - `api_runtime_mode()`
  - `api_runtime_llm_config()`

规则如下：

1. 从开启切到关闭

- 调 `POST /api/runtime-mode`
- 后端执行 `set_runtime_llm_enabled(False)`
- 当前运行模式立即变成无 Key 逻辑模式
- 聊天助手禁用

2. 从关闭切到开启

- 先看 `has_saved_runtime_llm_config()`
- 如果当前已经有可复用配置
  - 直接开启
- 如果没有配置
  - 返回 `requires_config=True`
  - 前端显示运行模式下方的输入框
  - 用户填写 URL、Key、模型供应商、模型名
  - 提交到 `POST /api/runtime-llm-config`
  - 后端执行 `save_runtime_llm_config()`
  - 立即切到 LLM Agent 模式

这里的“已保存配置”既可能来自：

- 启动时 `.env` 里已有的配置
- 也可能来自当前服务进程里，用户刚刚在页面里提交过的一组运行时配置

### 2.3 LLM 已开启但调用失败时怎么回到重配置

如果当前页面已经处于 `LLM Agent 模式`，但后续请求碰到：

- 连接错误
- 鉴权错误
- provider 错误

后端会返回一份额外的提示数据：

- `show_runtime_config=True`
- `runtime_payload`
- `reason`

前端收到后会：

- 立即展开运行模式下方配置表单
- 让开关和表单都震动一次
- 自动滚动到运行模式区域
- 把错误原因写到提示文案里
- 暂存刚才失败的动作

用户重新保存一组可用的 URL、Key、模型供应商、模型名后，前端会自动重试这次失败请求。

### 2.4 前端怎么知道当前模式

前端有两套来源：

1. 首屏渲染

- `index.html` 里的 `initial_runtime`
- `window.__INITIAL_RUNTIME__`

2. 后续同步

- 页面启动后再调用 `/api/runtime-info`
- 开关切换后调用 `/api/runtime-mode`
- 提交配置后调用 `/api/runtime-llm-config`

后端在 `web/app.py` 的 `build_runtime_payload()` 里统一返回：

- 当前模式 `mode`
- 开关是否开启 `switch_checked`
- 是否已有可复用配置 `has_saved_llm_config`
- 是否需要显示输入框 `requires_config`
- 页面顶部文案
- token 提示
- 聊天助手是否可用

所以页面顶部的“无 Key 逻辑模式 / LLM Agent 模式”、开关状态、配置表单是否显示，都不是前端自己猜的，而是后端统一下发的运行时状态。

### 2.4 按运行模式看完整工作链路

这一节单独按“规则模式”和“LLM Agent 模式”来拆，是为了把同一个功能在两种模式下到底走了哪条链路、落到了哪个文件的哪个方法，讲得更直观。

#### 2.4.1 规则模式链路

规则模式的总入口判断在：

- `config.py` 的 `AppConfig.llm_enabled()`
- `web/app.py` 的 `runtime_llm_enabled()`
- `web/app.py` 的 `runtime_mode()`
- `web/app.py` 的 `build_runtime_payload()`

只要当前运行时没有启用可用的 LLM 配置，`web/app.py` 里的接口就会走规则分支。

1. 内容创作模块

前端链路：

- `web/static/app.js` 的 `runCreatorModule()`
- 请求 `POST /api/module-create`

后端链路：

- `web/app.py` 的 `api_module_create()`
- `web/app.py` 的 `build_seed_topic()` 先整理种子主题
- `main.py` 的 `run_topic()` 进入单 Agent 包装
- `graph.py` 的 `BilibiliAgentGraph.run_single_agent("topic", ...)`
- `agents/topic_agent.py` 的 `TopicAgent.run()`
- `agents/topic_agent.py` 内部继续调用：
  - `fetch_hot_videos()`
  - `fetch_partition_videos()`
  - `fetch_peer_up_videos()`
  - `_generate_seed_topics()`
  - `_generate_trending_topics()`
  - `_merge_ideas()`
- 回到 `web/app.py` 的 `build_creator_topic_result()`，把基础选题结果改写成更贴近创作者输入的结果
- `main.py` 的 `run_copy()`
- `graph.py` 的 `BilibiliAgentGraph.run_single_agent("copy", ...)`
- `agents/copywriting_agent.py` 的 `CopywritingAgent.run()`
- `agents/copywriting_agent.py` 内部继续调用：
  - `_fallback()`
  - `_build_titles()`
  - `_build_script()`
  - `_build_description()`
  - `_build_tags()`
  - `_build_pinned_comment()`
- 最终由 `web/app.py` 的 `api_module_create()` 返回 `topic_result` 和 `copy_result`

这一条规则链路的本质是：

- 先用 `TopicAgent` 做样本抓取和启发式选题
- 再用 `CopywritingAgent` 做结构化文案生成
- 中间由 `web/app.py` 做创作者语境适配

2. 视频分析模块

前端链路：

- `web/static/app.js` 的 `scheduleVideoResolve()` / `resolveVideoLink()` 先请求 `POST /api/resolve-bili-link`
- `web/static/app.js` 的 `runAnalyzeModule()` 再请求 `POST /api/module-analyze`

视频预解析链路：

- `web/app.py` 的 `api_resolve_bili_link()`
- `web/app.py` 的 `extract_bvid()`
- `web/app.py` 的 `fetch_video_info()`
- `web/app.py` 的 `fetch_video_info_via_public_api()`
- 如果失败再回退到：
  - `bilibili_api.video.Video(...).get_info()`
  - `fetch_video_info_via_html()`
- 最后由 `build_resolved_payload()` 输出统一 `resolved` 结构

规则分析链路：

- `web/app.py` 的 `api_module_analyze()`
- 如果前端带来的 `resolved` 还能复用，就先用 `is_resolved_payload_usable()` 判断是否仍有效
- `web/app.py` 的 `classify_video_performance()` 做规则表现判断
- `main.py` 的 `run_topic()`，继续走 `TopicAgent.run()`，用于生成延展选题和对标样本
- `main.py` 的 `run_optimize()`，继续走 `OptimizationAgent.run()`
- `agents/optimization_agent.py` 内部继续调用：
  - `fetch_video_metrics()`
  - `_estimate_completion_rate()`
  - `_rule_based_diagnosis()`
  - `db.py` 的 `SQLiteStore.save_video_metrics()`
- 如果判断为热门视频：
  - `web/app.py` 的 `build_hot_analysis()`
- 如果判断为低表现视频：
  - `main.py` 的 `run_copy()`
  - `agents/copywriting_agent.py` 的 `CopywritingAgent.run()`
  - `web/app.py` 的 `build_low_performance_analysis()`
- 参考视频部分由：
  - `select_reference_videos()`
  - `build_reference_query_text()`
  - `build_reference_rank_entry()`
  - `build_reference_videos_from_market_snapshot()`

这一条规则链路的本质是：

- 先把视频公开数据解析成统一结构
- 再用启发式规则判断表现
- 最后补选题、优化建议和必要时的新文案

3. 智能对话助手

规则模式下不会启用聊天能力，对应控制点在：

- `web/app.py` 的 `api_chat()`
- `web/app.py` 的 `runtime_llm_enabled()`

也就是说，规则模式下聊天面板只是前端展示存在，后端能力是关闭的。

#### 2.4.2 LLM Agent 模式链路

LLM Agent 模式下，入口判断仍然是：

- `config.py` 的 `AppConfig.llm_enabled()`
- `web/app.py` 的 `runtime_llm_enabled()`
- `web/app.py` 的 `runtime_mode()`

区别在于 `web/app.py` 的几个核心接口会切到 `run_llm_*` 系列方法。

1. 内容创作模块

前端链路不变：

- `web/static/app.js` 的 `runCreatorModule()`
- 请求 `POST /api/module-create`

后端链路改成：

- `web/app.py` 的 `api_module_create()`
- `web/app.py` 的 `run_llm_module_create()`
- `web/app.py` 的 `get_llm_workspace_agent()`
- `agents/llm_workspace_agent.py` 的 `LLMWorkspaceAgent.run_structured()`

内容创作模式下注册和允许使用的工具是：

- `web/app.py` 的 `get_llm_workspace_agent()` 注册 `creator_briefing`
- `run_llm_module_create()` 调用 `run_structured(..., allowed_tools=["creator_briefing"], required_tools=["creator_briefing"])`

工具实际落点是：

- `web/app.py` 的 `build_creator_briefing()`
- `build_creator_briefing()` 内部会调用：
  - `build_seed_topic()`
  - `build_market_snapshot()`
  - `TopicAgent.fetch_hot_videos()`
  - `TopicAgent.fetch_partition_videos()`
  - `TopicAgent.fetch_peer_up_videos()`
- 返回市场样本和用户输入整理结果给 Agent

Agent 产出结果后，后端还会继续做：

- `normalize_copy_result_payload()` 统一清洗 `copy_result`
- 如果 Agent 中枢失败，则回退到 `run_llm_module_create_fallback()`
- `run_llm_module_create_fallback()` 会直接调用 `LLMClient.invoke_json_required()`

所以 LLM 模式下的内容创作不是直接把用户输入丢给模型，而是：

- 先通过 `creator_briefing` 把市场样本抓回来
- 再由 `LLMWorkspaceAgent` 基于工具结果产出结构化 JSON
- 最后由后端再次清洗和兜底

2. 视频分析模块

前端链路仍然一样：

- `resolveVideoLink()` 先拿 `resolved`
- `runAnalyzeModule()` 再发 `POST /api/module-analyze`

后端链路改成：

- `web/app.py` 的 `api_module_analyze()`
- 如果需要，仍先走：
  - `resolve_video_payload()`
  - `extract_bvid()`
  - `fetch_video_info()`
  - `build_resolved_payload()`
- 然后构造市场快照：
  - `build_market_snapshot()`
- 再进入：
  - `run_llm_module_analyze()`
  - `get_llm_workspace_agent()`
  - `LLMWorkspaceAgent.run_structured()`

视频分析模式下允许的工具是：

- `run_llm_module_analyze()` 里的 `allowed_tools=["hot_board_snapshot"]`

工具实际落点是：

- `web/app.py` 的 `build_hot_board_snapshot()`
- `build_hot_board_snapshot()` 内部继续调用 `build_market_snapshot()`

Agent 完成后，后端还会继续做：

- `normalize_performance_payload()`
- 如果有 `copy_result`，再走 `normalize_copy_result_payload()`
- `build_reference_videos_from_market_snapshot()` 构造参考视频

如果 Agent 中枢失败，还会回退到：

- `run_llm_module_analyze_fallback()`
- 最终由 `LLMClient.invoke_json_required()` 直接产出 JSON

所以 LLM 模式下的视频分析，本质上是：

- 视频真实信息仍由后端先解析好
- 市场样本仍由后端先准备好
- LLM 主要负责在这些结构化输入上做判断、分析和建议生成

3. 智能对话助手

这个功能只在 LLM Agent 模式下存在。

前端链路：

- `web/static/app.js` 的 `sendAssistantMessage()`
- 请求 `POST /api/chat`

后端链路：

- `web/app.py` 的 `api_chat()`
- `web/app.py` 的 `run_llm_chat()`
- `get_llm_workspace_agent()`
- `LLMWorkspaceAgent.run_structured()`

聊天模式下允许的工具是：

- `creator_briefing`
- `video_briefing`
- `hot_board_snapshot`

这些工具对应的实现方法分别是：

- `build_creator_briefing()`
- `build_video_briefing()`
- `build_hot_board_snapshot()`

其中：

- `build_video_briefing()` 会继续调用 `extract_bvid()`、`fetch_video_info()`、`build_llm_video_payload()`、`build_market_snapshot()`
- `build_creator_briefing()` 会继续调用 `build_seed_topic()` 和 `build_market_snapshot()`

聊天结果返回前，后端还会调用：

- `extract_reference_links_from_tool_observations()`

也就是说，聊天助手的本质不是单纯问答，而是：

- 先由 `LLMWorkspaceAgent` 判断要不要调工具
- 再用工具结果组织结构化答案
- 最后把参考视频链接也一起补给前端

## 3. 功能一：内容创作模块是怎么实现的

这个功能对应前端“内容创作”面板。

用户输入：

- 领域
- 方向
- 想法
- 分区
- 文案风格

前端点击“一键生成选题与文案”后，`web/static/app.js` 会请求：

```text
POST /api/module-create
```

### 3.1 规则模式下的实现链路

规则模式下，`web/app.py` 的 `api_module_create()` 会做 4 件事：

1. 先把用户输入整理成一个更适合分析的 `seed_topic`
2. 调 `run_topic()` 跑选题 Agent
3. 调 `build_creator_topic_result()` 把规则结果进一步改造成“创作者视角”的选题结果
4. 再调 `run_copy()` 生成完整文案

也就是：

```text
用户输入
-> build_seed_topic()
-> TopicAgent.run()
-> build_creator_topic_result()
-> CopywritingAgent.run()
-> 返回前端
```

### 3.2 选题功能的核心原理

选题能力主要在 `agents/topic_agent.py`。

它不是凭空猜题，而是先抓样本，再打分，再产出方向。

#### 1. 先抓 3 类样本

`TopicAgent.run()` 会拉三类数据：

- 全站热点：`fetch_hot_videos()`
- 分区样本：`fetch_partition_videos()`
- 同类 UP 样本：`fetch_peer_up_videos()`

这些数据最终都会被统一成 `VideoMetrics`。

#### 2. 把样本统一成一套可比较指标

`_build_metrics()` 会把不同来源的数据都转成统一字段：

- 播放
- 点赞
- 投币
- 收藏
- 评论
- 分享
- 时长
- 点赞率
- 估算平均观看时长
- 估算完播率

这里有一个重要原则：

- B 站公开接口不一定直接给完播率，所以代码会根据互动强度估算 `avg_view_duration` 和 `completion_rate`
- 这是启发式估算，不是官方真实完播率

#### 3. 计算竞争度和综合分

`_competition_scores()` 会按标题关键词把视频分桶，粗略估算同题材竞争度。

`_score_video()` 的核心思想是：

- 播放越高越好
- 点赞率、完播率越高越好
- 同类竞争越低越好

综合分大致由三部分组成：

- 流量规模
- 互动质量
- 竞争惩罚

所以它选题时不是只看播放量，而是看“流量 + 互动 + 竞争”的平衡。

#### 4. 有种子主题时，优先做“围绕当前想法的变体”

如果用户已经输入了明确想法，`_generate_seed_topics()` 不会直接丢掉，而是：

- 先清洗主题文本
- 从样本里找相关视频
- 识别主题模式

当前项目里内置了几种常见模式：

- 第一条视频做什么
- 开场钩子怎么设计
- 系列内容怎么排
- 舞蹈/颜值类第一条视频的人设开场
- 泛化主题延展

也就是说，系统不是只会给“热点榜同款”，而是尽量围绕用户自己的方向做可执行变体。

#### 5. 最终只保留 3 个最适合前端展示的方向

`_merge_ideas()` 会合并：

- 用户种子主题优先结果
- 热点趋势结果

最后保留 3 条，避免前端给太多但不聚焦的建议。

### 3.3 文案生成的核心原理

文案能力在 `agents/copywriting_agent.py`。

它的核心不是“让模型自由写一段话”，而是先做结构识别，再生成结构化结果。

#### 1. 先识别这个主题属于哪类表达场景

`_topic_mode()` 会判断当前主题属于：

- `first_video`
- `opening_hook`
- `series_plan`
- `dance_first_video`
- `general`

这一步很关键，因为不同主题的文案结构完全不一样。

比如：

- “第一条视频拍什么”更强调切口选择
- “开场怎么留人”更强调前三秒和镜头顺序
- “做系列内容”更强调多条内容分工

#### 2. 再抽出主题主体

`_extract_subject()`、`_account_subject()`、`_content_subject()` 会把题目里的主体抽出来。

目的不是做 NLP 炫技，而是为了让生成结果更自然：

- 标题能写得像真实 UP 主会用的话
- 脚本段落能围绕同一个主体展开
- 标签不会全部重复大词

#### 3. 先准备一份可直接落地的规则兜底文案

`_fallback()` 会直接生成：

- 3 个标题
- 至少 4 段脚本
- 简介
- 标签
- 置顶评论

这里的意义非常实际：

- 就算 LLM 不可用，功能也能返回完整结果
- 就算 LLM 返回格式不对，也能回退到稳定结构

#### 4. 有 Key 时再让 LLM 优化，但输出必须满足 JSON 结构

`run()` 会调用 `LLMClient.invoke_json()`，要求模型返回固定字段：

- `titles`
- `script`
- `description`
- `tags`
- `pinned_comment`

如果模型输出不合格，就回退到规则兜底。

所以这个模块的原则不是“模型写什么都收”，而是：

- 规则模板保证可用
- LLM 只是在可控结构上做增强

### 3.4 为什么 `build_creator_topic_result()` 还要二次加工

因为 `TopicAgent` 更偏“公共热点样本分析”，而内容创作页面更偏“创作者我要怎么做”。

所以 `web/app.py` 里又做了一层改写：

- 把用户领域、方向、想法进一步归一化
- 提取趋势关键词
- 根据分区补上常见表达角度
- 重新组织成更像创作者会直接拿去拍的视频题目

这一步的作用是把“市场样本结果”翻译成“创作者可执行结果”。

## 4. 功能二：视频分析模块是怎么实现的

这个功能对应前端“视频分析”面板。

用户输入：

- 一个 B 站视频链接

这个模块实际上分成两步：

1. 自动解析视频信息
2. 在解析结果基础上做表现判断、原因分析和优化建议

### 4.1 链接自动解析是怎么做的

前端在用户输入链接后，不会立刻分析，而是先做一次 550ms 防抖，再请求：

```text
POST /api/resolve-bili-link
```

对应 `web/static/app.js` 里的 `scheduleVideoResolve()` 和 `resolveVideoLink()`。

后端解析链路在 `web/app.py`：

```text
extract_bvid()
-> fetch_video_info()
-> build_resolved_payload()
```

#### 1. 先提取 BV 号

`extract_bvid()` 会：

- 识别常规链接里的 BV 号
- 如果是 `b23.tv` 这类短链，先展开再提取

#### 2. 再按“多级回退”拿视频信息

`fetch_video_info()` 的顺序是：

1. 官方公开接口 `x/web-interface/view`
2. `bilibili_api`
3. 页面 HTML 解析

这表示项目在设计上不是依赖单一路径，而是优先做“容错”。

#### 3. 把原始视频信息转换成前端可用的统一结构

`build_resolved_payload()` 会补齐：

- `bv_id`
- `up_name`
- `partition`
- `partition_label`
- `title`
- `topic`
- `style`
- `stats`

这里还有两步推断：

- `map_partition()`：把 B 站分区映射成项目内部统一分区
- `guess_style()`：根据标题和分区猜测内容风格

这一步的目的，是为后续分析、选题延展、文案重写提供统一上下文。

### 4.2 规则模式下，视频表现是怎么判断的

规则模式下，`api_module_analyze()` 会先调用：

- `classify_video_performance()`

它不是用单一阈值判断，而是做一个组合评分。

#### 1. 评分使用的核心指标

主要看四类信号：

- 播放量
- 收藏率 / 收藏量
- 点赞率
- 投币率

#### 2. 为什么收藏权重比较高

当前项目的判断逻辑里，收藏代表“留存价值”和“复用价值”。

对 B 站内容来说：

- 高播放但低收藏，可能只是被点进来
- 高收藏通常说明内容有明显可保留价值

所以规则里收藏率的加权比较重。

#### 3. 最终怎么得到“热门爆款 / 播放偏低”

综合分会被归一到一个范围内，最后用阈值判断：

- 分数较高，归类为“热门爆款”
- 否则归类为“播放偏低”

这一步返回的不是只有一个标签，还会附带 `reasons`，把加分或减分原因写出来。

### 4.3 热门视频和低表现视频为什么走不同分析路径

因为优化目标不同。

#### 热门视频

调用 `build_hot_analysis()`。

重点不是“补救”，而是：

- 这个视频为什么能火
- 标题强在哪里
- 后续还能沿着什么方向继续放大

所以输出偏“拆爆款逻辑 + 延展选题”。

#### 低表现视频

调用 `build_low_performance_analysis()`。

重点变成：

- 现有视频哪里弱
- 标题怎么改
- 封面怎么改
- 内容结构怎么改
- 接下来换什么题目更稳

所以输出偏“诊断 + 修复”。

### 4.4 优化建议是怎么生成的

优化能力在 `agents/optimization_agent.py`。

它做三件事：

1. 拉当前视频指标
2. 写入 SQLite 做历史留存
3. 结合基准样本生成诊断和建议

#### 1. 当前视频指标怎么来

`fetch_video_metrics()` 会通过 `bilibili_api.video.Video(...).get_info()` 拉取视频信息。

然后统一成 `VideoMetrics`，并估算：

- 完播率
- 平均观看时长

#### 2. 为什么要写入 SQLite

`SQLiteStore.save_video_metrics()` 会把每次分析时的视频快照写到 `video_metrics` 表。

这样后续优化分析可以看到：

- 当前视频历史上被分析过多少次
- 最近几次快照是什么样

也就是说，数据库不是业务主存储，而是“分析快照仓库”。

#### 3. 规则诊断怎么做

`_rule_based_diagnosis()` 会把当前视频和同类基准样本对比。

主要比较：

- 当前播放是否明显低于同类
- 完播率是否偏低
- 点赞率是否偏低

再输出对应建议，比如：

- 前 3 秒先给结果
- 减少铺垫
- 增加可收藏清单

这套逻辑的本质是“把内容问题拆成标题、开场、节奏、收藏价值几个维度”。

#### 4. 为什么优化 Agent 里也会调 LLM

`OptimizationAgent.run()` 会先准备一份规则 fallback，然后再调用 `LLMClient.invoke_json()`。

目的不是让 LLM 从零分析，而是：

- 规则先给出稳定底稿
- LLM 在底稿基础上润色诊断和建议

所以这个模块和文案模块一样，都是“规则保底 + LLM 增强”。

### 4.5 视频分析为什么还会生成新文案

因为当前模块不只是判断视频好坏，还要给“下一步怎么改”。

所以规则模式下：

- 如果是热门视频，不会强制重写文案
- 如果是低表现视频，会再调一次 `run_copy()` 给出一版新的标题/脚本/简介/标签/置顶评论

也就是：

- 热门：重点是拆解
- 低表现：重点是修复

### 4.6 参考视频是怎么来的

在视频分析模块里，后端还会尽量返回“可以直接点开的参考视频”。

实现思路是：

- 从市场样本里筛真实 BV 视频
- 组合标题、主题、分区作为检索词
- 做相关视频筛选和排序
- 如果候选视频缺少明确点赞数，再补一次公开视频详情
- 只把播放量和点赞数都明确的卡片返回给前端

这样前端展示的参考视频，不是静态示例，而是当前分析上下文下动态拼出来的结果。

## 5. 功能三：智能对话助手是怎么实现的

这个功能只在有 `LLM_API_KEY` 时开启。

前端会把下面这些信息一起发给后端：

- 当前用户消息
- 聊天历史
- 内容创作面板里的上下文
- 视频链接输入框里的上下文

对应接口：

```text
POST /api/chat
```

### 5.1 为什么聊天功能不是普通问答

因为后端不是直接把用户消息丢给模型，而是走 `run_llm_chat()`，内部会调用 `LLMWorkspaceAgent.run_structured()`。

这个 Agent 的设计原则很明确：

- 模型不能乱调用不存在的工具
- 模型不能直接返回一大段散文式输出
- 模型必须在“工具调用”和“最终结果”之间二选一
- 最终结果必须满足固定 JSON 结构

### 5.2 `LLMWorkspaceAgent` 的工作原理

`agents/llm_workspace_agent.py` 实现的是一个“受限多步 Agent”。

每一轮它都让模型返回一个 JSON：

- 要么 `action=某个工具名`
- 要么 `action=final`

如果是工具：

- 后端执行对应 `handler`
- 把结果追加到 `scratchpad`
- 进入下一轮

如果是最终结果：

- 校验必须字段是否齐全
- 校验要求调用的工具是否已经调用过
- 通过后才返回给上层

也就是说，这不是 ReAct 风格的松散实现，而是一个严格受控的 JSON Agent 循环。

### 5.3 聊天助手能调用哪些工具

在 `web/app.py` 的 `get_llm_workspace_agent()` 里注册了 3 个工具：

1. `creator_briefing`
   - 根据领域、方向、想法、分区抓创作样本
2. `video_briefing`
   - 根据视频链接抓真实视频信息和市场样本
3. `hot_board_snapshot`
   - 获取分区热点和样本快照

这意味着聊天助手虽然看起来像自由对话，但本质上仍然是“围绕项目已有业务能力来回答”。

### 5.4 聊天结果为什么还能附带参考链接

聊天接口返回后，后端会根据工具观测结果提取参考视频链接，再一起返回。

所以用户问：

- “帮我分析这个视频”
- “最近这个方向什么内容火”

聊天面板里除了文本回复，还可能出现可点击的视频参考卡片。

### 5.5 语音输入是怎么实现的

前端 `web/static/app.js` 里用了浏览器原生 `SpeechRecognition / webkitSpeechRecognition`。

也就是说：

- 语音识别发生在浏览器
- 后端收到的仍然是文本
- 后端本身不做语音识别

## 6. 功能四：运营 Agent 和全流水线是怎么实现的

虽然当前 Web 主界面重点是“内容创作”和“视频分析”，但项目底层还保留了完整的 4 Agent 流水线。

### 6.1 运营 Agent 做什么

`agents/operation_agent.py` 负责评论互动建议。

核心逻辑：

- 拉视频评论
- 判断是否垃圾评论
- 生成回复建议
- 生成点赞建议
- 识别高价值用户后给出关注建议

### 6.2 为什么默认是 dry-run

`process_video_interactions()` 返回的是“建议动作”，默认不直接真实执行。

原因很实际：

- 避免误删
- 避免误回
- 避免自动化操作过重

所以这个模块当前更像“运营建议引擎”，而不是“全自动操作机器人”。

### 6.3 全流水线怎么串起来

`graph.py` 里的 `BilibiliAgentGraph` 使用 LangGraph 把 4 个节点按顺序串起来：

```text
topic -> copy -> operate -> optimize
```

每个节点都读取/更新同一个 `PipelineState`。

这表示项目底层设计不是单独几个零散脚本，而是已经具备“可编排工作流”的结构。

### 6.4 CLI 为什么和 Web 共用同一套能力

`main.py` 只是把这些能力封装成：

- `run_topic`
- `run_copy`
- `run_operate`
- `run_optimize`
- `run_pipeline`

Web 层直接复用这些函数。

好处是：

- CLI 和 Web 不会维护两套业务逻辑
- Agent 行为更一致
- 后续如果加 API 或任务调度，也能继续复用

## 7. LLM 调用层是怎么保证稳定性的

`llm_client.py` 不是简单包了一层 SDK，而是做了 4 件事。

### 7.1 统一模型入口

当前通过 `ChatOpenAI` 接入模型，配置来自 `.env`：

- `LLM_PROVIDER`
- `LLM_API_KEY`
- `LLM_BASE_URL`
- `LLM_MODEL`

### 7.2 错误分类

`classify_llm_error()` 会把错误归类成：

- 认证问题
- 限流
- 503
- 502/504
- 超时
- 配额问题

这样 Web 层就能区分：

- 是直接失败
- 还是应该重试
- 还是应该跳过同 provider fallback

### 7.3 自动重试

`_invoke_messages()` 会对可重试错误做有限次数重试。

这可以减少瞬时网关错误或短时限流带来的影响。

### 7.4 强制提取 JSON

`_extract_json_text()` 会从模型输出里尽量提取合法 JSON。

这是因为实际模型经常会：

- 外面包 Markdown
- 前后多写解释
- JSON 前后夹杂多余文本

当前项目通过“提取 JSON + fallback”的方式，把这类问题收敛掉了。

## 8. 数据层和数据结构是怎么配合的

### 8.1 `models.py` 的作用

`models.py` 定义了统一数据结构：

- `VideoMetrics`
- `TopicIdea`
- `CopywritingResult`
- `InteractionAction`
- `OperationResult`
- `OptimizationSuggestion`

这样各层传数据时不是随便拼 dict，而是先有统一语义。

### 8.2 为什么还需要 `to_plain_data()`

因为：

- Agent 内部适合用 dataclass
- Web 返回 JSON 时适合用普通 dict/list

`to_plain_data()` 就是做这个转换。

### 8.3 SQLite 在项目里的定位

`db.py` 当前只负责视频指标快照，不承担全部业务状态管理。

所以 SQLite 在这个项目里更像：

- 分析留痕
- 历史对比基础

而不是完整业务数据库。

## 9. 前端层的实现原理

### 9.1 页面为什么看起来像单页应用

虽然技术栈是 Flask + 原生 JS，不是 React/Vue，但 `web/static/app.js` 已经做了典型前端状态管理：

- `state` 保存当前页面状态
- `requestJson()` 统一请求后端
- `creatorResult()` / `videoResult()` 负责结果渲染
- `loadRuntimeInfo()` 控制运行模式展示

所以它本质上是一个轻量 SPA。

### 9.2 为什么输入视频链接后会先看到“视频信息预览”

因为前端把：

- 链接解析
- 深度分析

拆成了两步。

好处是：

- 用户先确认解析到的是不是自己这条视频
- 后端后续分析可以复用已经解析好的 `resolved` 结果
- 减少重复请求

### 9.3 为什么页面体验会比普通表单好一点

前端还做了这些增强：

- 防抖解析视频链接
- 进度条动画
- 一键复制标题/脚本/标签
- 封面加载失败重试
- 聊天回复打字机效果
- 浏览器语音输入

这些不改变业务逻辑，但明显改善了交互体验。

## 10. 当前项目的几个关键设计原则

把整个项目总结下来，最关键的是下面几条。

### 10.1 规则优先，LLM 增强

项目不是把所有能力都建立在 LLM 可用的前提上。

相反：

- 没有 Key 也能跑核心功能
- 有 Key 时再切到更强的 Agent 模式

这保证了基础可用性。

### 10.2 统一数据结构，再做多模块复用

不管是 Web、CLI、Agent 还是数据库，底层都围绕统一数据结构在转。

这样模块之间更容易拼装。

### 10.3 公开数据先结构化，再做判断

不管是选题还是视频分析，第一步都不是生成，而是先把公开数据转成统一指标，再做评分、归类、延展。

### 10.4 LLM 不是“随便聊”，而是受控执行

`LLMWorkspaceAgent` 的价值就在这里：

- 限定工具
- 限定步骤
- 限定输出结构
- 校验最终结果

这样比直接把任务发给模型更稳定。

### 10.5 失败时优先回退，不让功能直接废掉

当前项目大量使用 fallback：

- 视频信息解析多级回退
- 文案生成规则兜底
- 优化建议规则兜底
- LLM Agent 失败时回退到直调 LLM

这保证了系统在不稳定环境下仍然尽量给结果。

## 11. 一句话总结

当前项目本质上是一个“B 站创作与分析工作台”：

- 前端负责把创作和分析场景拆成清晰操作面板
- 后端负责把公开数据、规则判断、LLM 工具调用串成稳定链路
- 无 Key 时走规则引擎
- 有 Key 时切到严格受控的 LLM Agent
- 所有功能最终都围绕“选题、文案、分析、优化、互动建议”这几类能力展开

如果你后面还要继续扩展，这份项目最值得保留的不是某个 prompt，而是现在这套：

- 统一数据结构
- 明确功能分层
- 规则兜底
- Agent 受控调用

这几条骨架。
