# 项目说明

## 1. 项目定位

这是一个面向 B 站创作者的内容工作台，目标不是做一个泛泛的 AI Demo，而是把创作前和发布后的两段高频工作流拆开：

1. 还没发视频时，先做选题和文案
2. 已经发了视频后，做视频解析、表现判断和优化
3. 在配置 LLM 后，直接通过对话式助手发起任务

当前产品形态已经稳定为：

- 左侧 2 个业务模块
- 右侧 1 个智能对话助手
- 顶部 1 个运行模式与全局状态区

## 2. 当前核心能力

### 2.1 内容创作模块

适合：

- 还没有明确选题
- 只有模糊方向和想法
- 想直接得到标题、脚本、简介、标签、置顶评论

输入：

- 领域 `field`
- 方向 `direction`
- 想法 `idea`
- 内容分区 `partition`
- 文案风格 `style`

输出：

- 整理后的创作方向 `normalized_profile`
- 当前核心问题 `seed_topic`
- 3 个推荐选题 `topic_result.ideas`
- 推荐主选题 `chosen_topic`
- 一整套文案 `copy_result`

### 2.2 视频分析模块

适合：

- 已经发布了视频
- 想判断当前更像热门爆款还是播放偏低
- 想知道下一步该改标题、封面、结构还是重做选题

输入：

- B 站视频链接

输出：

- 自动解析后的视频结构化信息 `resolved`
- 表现判断 `performance`
- 原因分析 `analysis`
- 优化建议 `optimize_result`
- 后续可做选题 `topic_result`
- 可点击参考视频 `reference_videos`
- 低表现时额外返回一套新文案 `copy_result`

### 2.3 智能对话助手

适合：

- 不想自己判断该用哪个模块
- 想直接说“帮我分析这个视频”“帮我选题”“帮我写文案”
- 希望系统自己决定先调用哪个工具

输入：

- 自然语言问题 `message`
- 最近对话历史 `history`
- 页面上下文 `context`

输出：

- 直接回答 `reply`
- 下一步建议 `suggested_next_actions`
- 工具观察结果提炼出的参考视频 `reference_links`

## 3. 双模式运行逻辑

### 3.1 规则模式

触发条件：

- `.env` 里没有配置 `LLM_API_KEY`

特点：

- Web 主工作流走纯代码规则链路
- 智能助手关闭
- 不消耗 LLM token
- 页面顶部会显示“无 Key 规则模式”

### 3.2 LLM Agent 模式

触发条件：

- `.env` 里配置了 `LLM_API_KEY`

特点：

- `module-create` 和 `module-analyze` 切到 `LLMWorkspaceAgent`
- 助手启用
- 分析、判断、文案生成由大模型完成
- 会消耗 LLM token
- 页面顶部会显示“LLM Agent 模式”

### 3.3 重要边界

主界面是双模式切换，但兼容接口仍然保留：

- `/api/topic`
- `/api/copy`
- `/api/operate`
- `/api/optimize`
- `/api/pipeline`

这些兼容接口并不走 `LLMWorkspaceAgent` 中枢，但在配置了 `LLM_API_KEY` 后，其中部分 Agent 仍然会直接调用 `LLMClient`。

## 4. 当前代码结构

可以把项目理解成 4 层：

```text
web/app.py                -> Web 入口、接口编排、运行模式切换
main.py / graph.py        -> CLI 封装与流程编排
agents/*.py               -> 业务能力实现
config.py / db.py / models.py / llm_client.py -> 配置、存储、模型、LLM 客户端
```

当前主要文件职责如下：

- `web/app.py`
  负责 Flask 路由、视频解析、运行模式信息、LLM 工具注册、模块级编排、参考视频检索
- `web/templates/index.html`
  负责页面结构
- `web/static/app.js`
  负责前端交互、自动解析、进度条、复制、聊天、语音输入
- `web/static/style.css`
  负责页面样式和响应式布局
- `graph.py`
  用 LangGraph 串联 `topic -> copy -> operate -> optimize`
- `main.py`
  封装 CLI 和可复用函数入口
- `agents/topic_agent.py`
  抓取热点、分区样本、同类 UP 样本，输出选题
- `agents/copywriting_agent.py`
  生成标题、脚本、简介、标签、置顶评论
- `agents/optimization_agent.py`
  拉取视频指标、输出优化建议
- `agents/operation_agent.py`
  处理评论互动建议
- `agents/llm_workspace_agent.py`
  作为有 Key 模式下的 LLM Agent 中枢
- `llm_client.py`
  对 `langchain_openai.ChatOpenAI` 的统一封装，负责 JSON 提取、超时、重试和错误分类
- `config.py`
  读取 `.env`
- `db.py`
  使用 SQLite 存储视频指标历史
- `models.py`
  定义数据模型和 dataclass

## 5. 当前 Web 接口

### 5.1 主界面核心接口

- `GET /api/runtime-info`
  返回运行模式、是否启用聊天、token 提示、切换提示
- `POST /api/resolve-bili-link`
  只解析视频链接，不做分析
- `POST /api/module-create`
  内容创作主入口
- `POST /api/module-analyze`
  视频分析主入口
- `POST /api/chat`
  智能对话入口，仅在有 Key 时可用

### 5.2 兼容接口

- `POST /api/topic`
- `POST /api/copy`
- `POST /api/operate`
- `POST /api/optimize`
- `POST /api/pipeline`

## 6. 新增或已存在但旧文档常漏掉的能力

下面这些能力在当前代码里已经存在，文档更新时必须计入：

- 视频链接粘贴后会先自动调用 `/api/resolve-bili-link`
- 支持常规链接和 `b23.tv` 短链
- 前端左侧是“顶部标签切换模块”，不是旧文档里的悬停伸缩双面板
- 聊天助手支持快捷提问按钮
- 浏览器支持时可用语音输入 `SpeechRecognition / webkitSpeechRecognition`
- 聊天回复使用打字机式逐字输出
- 结果区大量按钮支持一键复制
- 视频封面有懒加载、重试和 fallback 占位逻辑
- 视频分析结果会补充“可直接打开的参考视频”
- 参考视频不只来自本地样本，还会尝试：
  - 当前视频相关推荐
  - B 站搜索接口
- LLM 模式下有两层回退：
  - 先尝试 `LLMWorkspaceAgent`
  - 再尝试单次 direct LLM fallback
- 如果 fallback 成功，返回结果里会带 `llm_warning`
- LLM Agent 内部会保留：
  - `agent_trace`
  - `tool_observations`

## 7. 分区与风格逻辑

前端展示的分区选项很多，但后端最终会归一化到 5 个核心分区：

- `knowledge`
- `tech`
- `life`
- `game`
- `ent`

例如这些值会被归一化：

- `business` / `career` / `study` -> `knowledge`
- `ai` / `digital` / `auto` -> `tech`
- `food` / `vlog` / `emotion` / `fashion` / `pet` / `sports` -> `life`
- `beauty` / `dance` / `music` / `film` / `anime` -> `ent`

当前文案风格有 4 种：

- `干货`
- `教学`
- `搞笑`
- `混剪`

## 8. 数据来源

项目当前依赖的数据来源主要有三类：

### 8.1 B 站公开数据

- 热榜
- 分区样本
- 同类 UP 主视频
- 当前视频公开信息
- 当前视频相关推荐
- B 站搜索结果

### 8.2 本地 SQLite

- 文件默认是 `bilibili_agents.db`
- 主要存视频指标历史

### 8.3 LLM Provider

- 仅在 `.env` 配置 `LLM_API_KEY` 后使用
- 通过 `langchain-openai` 统一接入

## 9. 当前边界

这个项目当前强调：

- 可运行
- 可学习
- 可改造

但它不是：

- 多租户后台系统
- 高并发生产调度平台
- 完整的账号运营系统

另外要注意：

- B 站公开接口本身可能失效或被限流
- 某些 provider 对复杂 LLM 请求可能更慢
- `.env` 中的模型、超时和重试策略会直接影响体验

## 10. 适合谁

适合：

- B 站内容创作者
- 想学习 Agent 项目组织方式的开发者
- 想从 Java / SpringBoot 迁移到 Python Agent 项目的开发者

不太适合：

- 只想找一个现成的商用 SaaS 产品的人
- 需要企业级权限系统、审计系统和多租户体系的人
