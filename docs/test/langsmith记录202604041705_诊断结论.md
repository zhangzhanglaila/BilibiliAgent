## 问题摘要

这次测试使用的是一个播放量极低的钢琴视频，但轨迹里出现了与预期不符的信号：

1. LLM 思考中仍出现 `action: video_briefing`
2. `market_snapshot.peer_samples` 为空
3. 建议的直接对标视频与钢琴方向无关
4. 爆款样本的播放 / 点赞过滤可能没有稳定生效

## 诊断结论

这次问题的主因不是“当前代码真的又执行了一次 `video_briefing`”，而是“进入分析前的对标上下文不够具体”：

1. 钢琴 / 音乐方向没有足够早地被识别出来，导致 benchmark query 退化成宽泛词
2. `peer_samples` 在进入 LLM 时为空，模型自然会倾向继续规划补数据动作
3. `_safe_metric_int()` 没有完整覆盖 B 站的 `万 +`、小数万、亿、k 等格式，导致热门样本阈值过滤不稳定
4. Prompt 和动作校验器虽然已有约束，但没有把“优先引用同方向爆款样本、不要重新解析当前视频”压得足够明确

## 已完成修复

### 1. 同方向识别更具体

- 在 `web/app.py` 中新增并强化 `build_video_benchmark_profile()`
- 对音乐 / 钢琴内容提前识别
- 预加载 `benchmark_lane_label`、`benchmark_terms`、`benchmark_queries`

### 2. `peer_samples` 改为严格同方向爆款

- `build_hot_peer_market_snapshot()` 只预抓同方向爆款样本
- `fetch_hot_peer_videos()` 现在按以下规则过滤：
  - 近 `30` 天
  - 播放 `>= 5 万`
  - 点赞 `>= 1000`
  - 排除当前视频
  - 候选内容必须和 query 语义相关

### 3. 彻底压制“重新解析当前视频”的动作

- `VIDEO_ANALYZE_SYSTEM_PROMPT` 明确禁止 `video_briefing`
- `video_analyze_action_validator()` 对 `video_briefing`、`hot_board_snapshot` 做硬拦截
- retrieval 强制优先使用预加载的方向词和同方向爆款样本

### 4. B 站数值解析兼容补齐

- `_safe_metric_int()` 现已兼容：
  - `10 万 +`
  - `5.2 万`
  - `3.4亿`
  - `1.2k`

## 修复后预期

1. `peer_samples` 应该是纯同方向爆款，而不是泛赛道高播放噪音样本
2. 钢琴视频应优先拿到钢琴 / 音乐方向的爆款对标
3. LLM 分析链应明确引用“同方向爆款对标样本”，不再重复解析当前视频
4. 爆款样本过滤应稳定兼容 B 站所有常见展示格式
