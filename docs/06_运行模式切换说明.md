# 运行模式切换说明

这份文档只说明一件事：

- 当前项目进入页面时为什么是某个模式
- 页面里的模式开关到底改了什么

## 1. 首次进入页面时，默认模式怎么决定

当前默认状态由 `.env` 里的可用 LLM 配置决定。

关键入口：

- [config.py](../config.py)
- [web/app.py](../web/app.py) 的 `RUNTIME_LLM_ENABLED`
- [web/app.py](../web/app.py) 的 `RUNTIME_LLM_CONFIG`
- [web/app.py](../web/app.py) 的 `build_runtime_payload()`

判断逻辑：

- 如果环境里已经有可用 LLM 配置
  - 页面首次进入时默认可以是 LLM Agent 模式
- 如果环境里没有可用配置
  - 页面首次进入时是无 Key 逻辑模式

注意：

- 页面首屏状态是服务端渲染出来的
- 不是前端先写死，再异步修正

## 2. 页面里的开关控制什么

页面中的运行模式开关，控制的是：

- 当前这次 Web 工作台请求到底走规则链路还是 `LLMWorkspaceAgent`

它影响：

- 内容创作模块
- 视频分析模块
- 智能助手

它不直接影响：

- CLI
- 知识库接口
- 兼容接口的默认 Agent 实例

## 3. 从开启切到关闭

发生的事：

- Web 当前模式切回 `无 Key 逻辑模式`
- 智能助手立即禁用
- 后续内容创作与视频分析改走规则链路

对应接口：

- `POST /api/runtime-mode`

后端行为：

- `set_runtime_llm_enabled(False)`

注意：

- 关闭开关不会删除已保存配置
- 只是停用当前 Web 运行时的 LLM Agent

## 4. 从关闭切到开启

有两种情况。

### 4.1 已有已保存配置

结果：

- 直接切到 `LLM Agent 模式`

### 4.2 没有已保存配置

结果：

- 不会直接报错
- 前端会拉起运行时配置表单
- 用户填写后再保存

对应接口：

- `POST /api/runtime-mode`
- `POST /api/runtime-llm-config`

## 5. 当前保存的配置从哪里来

当前运行时配置来源有两种：

### 5.1 `.env`

如果服务启动时环境里有配置，页面首屏可直接拿到。

### 5.2 页面内填写

用户在运行模式区域填写：

- URL
- Key
- 模型供应商
- 模型名

保存后，后端把它存在运行时内存中，并立即切换到 LLM Agent 模式。

## 6. 当前模式信息包含哪些字段

`build_runtime_payload()` 当前会返回：

- `mode`
- `mode_label`
- `llm_enabled`
- `chat_available`
- `switch_checked`
- `has_saved_llm_config`
- `saved_config_source`
- `saved_provider`
- `saved_model`
- `saved_base_url`
- `saved_api_key_masked`
- `requires_config`
- `mode_title`
- `mode_description`
- `token_policy`
- `switch_hint`

这些字段会同时用于：

- 首屏渲染
- 切模式后前端刷新
- LLM 报错后拉起重配表单

## 7. 当前错误重配机制

如果 LLM 请求失败，后端会返回一份重配提示数据，前端会：

- 自动展开配置表单
- 把错误原因写进提示区
- 允许用户直接重新填写配置

这部分入口见：

- `build_llm_runtime_reconfigure_data()`

## 8. 当前最容易搞混的地方

### 8.1 页面模式不等于全项目模式

页面开关只管 Web 工作台。

### 8.2 页面关闭 LLM 不等于删除配置

配置还在，只是本次运行不用。

### 8.3 知识库功能不受这个开关影响

知识库上传、同步、查看、检索都不依赖这个开关。
